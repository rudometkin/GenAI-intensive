# PRD: AI-ассистент для онлайн-меню

**Версия:** 1.0  
**Дата:** 12 апреля 2026  
**Статус:** Draft

---

## 1. Обзор продукта

AI-ассистент для онлайн-меню — это плагин, который встраивается в QR-меню ресторанов и помогает гостям выбрать блюда. Ассистент учитывает пищевые ограничения, стиль питания, текущий запрос и уровень голода гостя и формирует персональную подборку из меню конкретного заведения.

Данная версия продукта является демонстрационным прототипом для потенциальных партнёров — сервисов QR-меню. Цель — показать механику встраивания, пользовательский опыт и ключевые преимущества решения.

---

## 2. Проблема

### 2.1 Контекст

В ресторане гость сталкивается с задачей выбора блюда без достаточной поддержки. Официанты не всегда могут дать качественную персональную рекомендацию: из-за загруженности, неопытности или просто отсутствия информации о предпочтениях конкретного гостя.

### 2.2 Ключевые сценарии проблемы

**Пищевые ограничения**  
Аллергики, люди с непереносимостями (лактоза, глютен и др.) и те, кто избирателен в еде по состоянию здоровья, вынуждены тщательно изучать каждое блюдо. Это занимает время и вызывает тревогу — особенно при серьёзных аллергиях.

**Контроль питания**  
Гости, которые следят за КБЖУ, считают калории или стараются потреблять достаточно белка, не могут быстро ориентироваться в меню без детального состава блюд.

**Незнакомая кухня**  
В ресторане с непривычной кухней (например, китайской или японской) даже фотография блюда не даёт понимания вкуса, состава и порции. Гость теряется и часто делает случайный выбор.

**Общий запрос "не знаю что взять"**  
Значительная часть гостей испытывает информационную перегрузку при взгляде на большое меню и хотела бы получить готовый совет под своё настроение и аппетит.

---

## 3. Решение

### 3.1 Механика

Плагин встраивается в существующие QR-меню партнёров. Партнёр получает код плагина и самостоятельно интегрирует его в свой продукт. Для гостя плагин выглядит как нативная часть меню — кнопка и чат-ассистент.

### 3.2 Как это работает

1. Гость открывает QR-меню ресторана.
2. Видит кнопку "Подбери себе блюдо" и тултип-подсказку.
3. Открывает чат и отвечает на несколько вопросов: что хочется сегодня, уровень голода, пожелания по напиткам.
4. При первом визите — дополнительно указывает аллергии и ограничения (один раз, они сохраняются).
5. Получает персональную подборку блюд с объяснением почему каждое блюдо подходит именно ему.
6. Переходит в карточку блюда и добавляет в корзину.

### 3.3 Технический подход (текущая версия)

Подбор блюд реализован на rule-based алгоритме: сопоставление предпочтений гостя с атрибутами блюд (ингредиенты, категория, калорийность, граммовка). В следующих версиях будет заменён на собственный AI-движок с поддержкой свободного диалога.

## 4. Целевая аудитория

| Сегмент | Описание | Ключевая потребность |
|---|---|---|
| Люди с пищевыми ограничениями | Аллергики, люди с непереносимостями, религиозные ограничения | Быстро найти безопасные блюда |
| Следящие за питанием | Считают КБЖУ, следят за белком, на диете | Фильтрация по нутриентам и составу |
| Гости в незнакомом ресторане | Впервые в заведении или в непривычной кухне | Навигация и объяснение блюд |

**Прямые партнёры (B2B):** сервисы QR-меню, которые интегрируют плагин в свой продукт.

## 5. Клиентский путь

### 5.1 Первый визит (нет сохранённых данных)

```
Открыл меню
    → Онбординг (один раз при первом визите)
    → Тултип на кнопку "Подбери себе блюдо"
    → Нажимает кнопку → открывается чат
    → [Шаг 1] Проверка программы лояльности
        → Есть аккаунт → авторизация по телефону + OTP
        → Нет аккаунта → продолжаем
    → [Шаг 2] Сессионные вопросы
        → Что хочется сегодня (мультивыбор)
        → Уровень голода
        → Пожелания по напиткам
    → Показываем рекомендации ← ГОСТЬ ПОЛУЧИЛ VALUE
    → [Шаг 3] Профильные вопросы (один раз)
        → "Есть аллергии или ограничения? Расскажем один раз — запомним"
        → Выбор из списка + поле "Другое"
        → Кнопка "Пропустить"
    → [Шаг 4] Предложение сохранить профиль
        → "Сохранить, чтобы не заполнять снова?"
        → Ввод телефона → OTP (можно пропустить)
    → Данные сохранены в localStorage (и в backend, если дал телефон)
```

### 5.2 Второй визит — тот же браузер (данные в localStorage)

```
Открыл меню
    → Тултип на кнопку
    → Нажимает кнопку → открывается чат
    → [Шаг 1] Проверка лояльности (если не авторизован ранее)
    → [Шаг 2] Только сессионные вопросы (2–3 шага)
        → Профильные вопросы пропускаются
    → Рекомендации с учётом сохранённого профиля
```

### 5.3 Второй визит — другое устройство или очищен кэш

```
→ Авторизован (есть аккаунт):
    Телефон → OTP → профиль подтягивается с backend
    → Только сессионные вопросы

→ Не авторизован:
    → Флоу первого визита полностью
```

## 6. Экраны и функциональные требования

### 6.1 Онбординг

**Назначение:** Подготовить гостя к тому, что в меню есть AI-помощник. Показывается один раз — при первом открытии меню.

**Содержимое экрана:**
- Анимированные иконки еды создают визуальный контекст ресторана
- Заголовок и короткое описание возможностей ассистента:
  - Подберёт блюда под вкус и ограничения
  - Запомнит предпочтения
  - Объяснит состав незнакомых блюд
- Кнопка "Открыть меню"

**Поведение:**
- Отображается поверх меню как полноэкранный оверлей
- Показывается ровно один раз; факт показа фиксируется в localStorage
- После нажатия кнопки — переход к экрану меню

---

### 6.2 Экран меню с тултипом

**Назначение:** Показать гостю, где находится кнопка ассистента и зачем она нужна.

**Тултип:**
- Появляется автоматически через 3 секунды после открытия меню
- Позиционируется рядом с кнопкой "Подбери себе блюдо"
- Текст: краткое описание функции кнопки
- Закрывается по тапу на тултип или по нажатию на кнопку
- Показывается один раз; факт показа фиксируется в localStorage

**Кнопка "Подбери себе блюдо":**
- Фиксированная, всегда видна на экране меню
- При нажатии открывает чат-ассистент
- Визуально выделена (основной акцентный цвет)

---

### 6.3 Чат-ассистент: Шаг 1 — Программа лояльности

**Назначение:** Идентифицировать гостя для персонализации и начисления бонусов.

**Вопрос ассистента:**
> "Есть ли у вас карта лояльности [название ресторана]?"

**Варианты ответа:**
- "Да, есть" → переход к авторизации
- "Нет" → переход к следующему шагу

**Если у ресторана нет программы лояльности:**
Этот шаг заменяется предложением зарегистрироваться для персонализации:
> "Хотите, чтобы мы запомнили ваши предпочтения? Укажите телефон — и не придётся заполнять анкету снова"
- Ввод телефона → OTP
- Кнопка "Пропустить"

---

### 6.4 Чат-ассистент: Авторизация (если есть аккаунт)

**Шаги:**
1. Поле ввода номера телефона
2. Запрос OTP-кода (одноразовый код, отправляется по SMS)
3. Поле ввода кода (6 цифр)
4. При успешной авторизации — приветствие по имени, если имя сохранено в профиле

**Поведение при ошибке:**
- Неверный код: сообщение об ошибке + возможность запросить код повторно
- Таймаут повторной отправки: 60 секунд

**После авторизации:**
- Если в профиле уже есть аллергии и предпочтения — профильные вопросы пропускаются
- Переход сразу к сессионным вопросам

---

### 6.5 Чат-ассистент: Шаг 2 — Сессионные вопросы

Задаются при каждом визите. Отражают текущее желание гостя, которое меняется от посещения к посещению.

#### Вопрос 2а: Что хочется сегодня?

**Тип:** мультивыбор, можно выбрать несколько вариантов

| Вариант | Иконка |
|---|---|
| Больше овощей | 🥗 |
| Мясо | 🥩 |
| Рыба | 🐟 |
| Морепродукты | 🦐 |
| Острое | 🌶️ |
| Суп | 🍲 |
| Сладкое | 🍰 |

#### Вопрос 2б: Уровень голода

**Тип:** одиночный выбор

| Вариант | Логика подбора |
|---|---|
| Совсем не голоден | Приоритет: салаты, лёгкие закуски, некалорийные блюда |
| Хочу перекусить | Приоритет: закуски, супы, небольшие порции |
| Голоден | Сбалансированный подбор |
| Очень голоден | Приоритет: основные блюда, большие порции, высококалорийные |

#### Вопрос 2в: Напитки

**Тип:** мультивыбор

| Вариант |
|---|
| Горячие напитки (кофе, чай) |
| Холодные напитки (соки, лимонады) |
| Алкоголь |
| Без напитков |

Запрашиваются после того, как гость уже получил рекомендации и убедился в ценности ассистента.

**Сообщение ассистента перед шагом:**
> "Есть ли у вас аллергии или продукты, которые вы не едите? Расскажите один раз — больше спрашивать не будем"

#### Вопрос 3а: Аллергии, непереносимости, нелюбимые продукты

**Тип:** мультивыбор + текстовое поле

| Вариант |
|---|
| Арахис |
| Орехи |
| Рыба |
| Яйца |
| Лактоза |
| Курица |
| Глютен |
| Кинза |
| Говядина |
| Свинина |

| Лук |

**Дополнительное действие:** кнопка "Пропустить"

**Поведение:**
- Выбранные ограничения сохраняются в профиль
- При последующих визитах блюда с этими ингредиентами исключаются из рекомендаций или помечаются предупреждением

---

### 6.7 Чат-ассистент: Шаг 4 — Сохранение профиля

Отображается только если гость ещё не авторизован.

**Сообщение ассистента:**
> "Хотите, чтобы мы запомнили ваши предпочтения? Тогда в следующий раз не придётся заполнять анкету"

**Сценарий А — гость даёт телефон:**
1. Поле ввода телефона
2. OTP-подтверждение
3. Профиль сохраняется в backend + localStorage

**Сценарий Б — гость пропускает:**
- Кнопка "Пропустить"
- Профиль сохраняется только в localStorage
- Данные будут доступны при следующем визите с того же устройства/браузера

---

### 6.8 Чат-ассистент: Рекомендации

**Назначение:** Показать подборку блюд, соответствующих ответам гостя.

**Состав подборки:**

Всегда включает напиток (если гость не выбрал "Без напитков"). Остальной набор формируется на основе ответов:

- Гость выбрал "суп" → в подборке суп
- Гость выбрал "сладкое" → в подборке десерт
- Уровень голода "очень голоден" → основное блюдо с большой порцией

**Карточка блюда в рекомендациях содержит:**
- Фотография блюда
- Название
- Персонализированное описание от ассистента — подчёркивает то, что важно именно этому гостю (например: *"Здесь много морепродуктов — как вы и хотели"*)
- Кнопка "В корзину" — добавляет блюдо в корзину на стороне партнёра
- Кнопка "Подробнее" — открывает полную карточку блюда

**Теги на карточках:**

| Тег | Условие показа |
|---|---|
| Большая порция | Граммовка выше среднего по категории |
| Некалорийно | Лёгкое блюдо, для невысокого уровня голода |
| Сытное блюдо | Калорийное, для высокого уровня голода |
| Соответствует вашим предпочтениям | Совпадение с выбором гостя в анкете |

---

### 6.9 Карточка блюда

Открывается из рекомендаций или из общего списка меню.

**Содержимое:**
- Полноразмерное фото блюда
- Название, граммовка, цена
- Описание блюда
- Персонализированный комментарий ассистента (если открыто из рекомендаций)
- Состав и аллергены
- Счётчик количества (кнопки −/+)
- Кнопка "Добавить в корзину"

---

### 6.10 Корзина

Корзина живёт на стороне партнёра. Плагин передаёт данные о выбранных блюдах в корзину партнёрского меню.

**Взаимодействие из плагина:**
- Кнопка "В корзину" в рекомендациях и карточке блюда добавляет позицию
- Изменение количества и удаление происходит в интерфейсе партнёра
- Оформление заказа — полностью на стороне партнёра

## 7. Хранение данных

### 7.1 Разделение данных

| Тип | Примеры | Хранилище |
|---|---|---|
| Профильные (долгосрочные) | Аллергии, непереносимости, стиль питания | localStorage + backend (если авторизован) |
| Сессионные (на один визит) | Что хочется сегодня, голод, напитки | Только в памяти (не сохраняются) |
| Идентификация | Номер телефона, имя | backend |

### 7.2 Профиль пользователя

```
{
  phone: string,          // опционально
  name: string,           // опционально
  birthday: string,       // опционально
  restrictions: string[], // аллергии и непереносимости
  savedAt: timestamp
}
```

### 7.3 Правила хранения

- Данные в localStorage сохраняются без срока истечения
- При авторизации localStorage синхронизируется с backend
- При конфликте (разные данные в localStorage и backend) приоритет у backend
- Удаление данных — по запросу пользователя (кнопка в профиле или через поддержку)

## 8. Интеграция с партнёрами

### 8.1 Механика встраивания

Плагин предоставляется партнёру в виде JavaScript-кода. Партнёр самостоятельно встраивает его в нужную часть своего интерфейса.

### 8.2 Что партнёр передаёт плагину

- Данные меню (список блюд с атрибутами)
- Конфигурацию программы лояльности (есть / нет, endpoint авторизации)
- Брендинг (цвета, название заведения)

### 8.3 Что плагин передаёт партнёру

- Выбранные блюда для добавления в корзину

### 8.4 Атрибуты блюда, необходимые для работы алгоритма

| Атрибут | Тип | Назначение |
|---|---|---|
| id | string | Идентификатор |
| name | string | Отображение |
| category | enum | Фильтрация по типу |
| ingredients | string[] | Сопоставление с предпочтениями |
| allergens | string[] | Исключение при ограничениях |
| weight | number | Определение размера порции |
| calories | number | Подбор по уровню голода |
| price | number | Отображение |
| image | url | Отображение |
| temperature | enum | Подбор напитков (горячее/холодное) |

---

## 9. Ограничения текущей версии

- Подбор блюд — rule-based, не учитывает контекст свободного диалога
- Нет backend-инфраструктуры (демо-версия работает локально)
- OTP-авторизация — заглушка, реальная отправка SMS не реализована
- Корзина интегрирована локально, без реального API партнёра
- Нет аналитики и admin-панели для ресторана
