# Инсайты: Kopirka

Ключевые наблюдения и выводы, полученные в процессе разработки проекта. Не являются документацией ошибок или решений — фиксируют общие закономерности и нетривиальные умозаключения.

---

## I-01 · Статический анализ бандлера переопределяет runtime-логику

Разработчики, пришедшие из webpack-экосистемы, интуитивно предполагают, что `try/catch` вокруг `import()` достаточен для опциональных зависимостей. Turbopack разрушает это предположение: граница между временем сборки и временем выполнения сдвигается влево, и код, некогда работавший в рантайме, теперь требует статической корректности.

Практический вывод: при переходе на Turbopack/Next.js 16 необходимо аудировать все условные импорты — любой `import()` с нестатическим путём или несуществующим модулем является потенциальной блокировкой сборки.

## I-02 · Качество контента требует итеративного тестирования на разных типах страниц

Алгоритм, демонстрирующий высокое качество на одном классе страниц (энциклопедические статьи), может полностью деградировать на другом (технические README, лонгриды, новостные сайты). Каждый новый тип страницы обнаруживал новую категорию артефактов:

- Wikipedia → библиографические цитаты в Summary
- GitHub README → фрагменты кода в Summary
- Плотно структурированные HTML → слипание предложений
- Статьи с вложенными изображениями-ссылками → пустые markdown-ссылки

Практический вывод: для систем извлечения и трансформации контента тест-покрытие должно включать репрезентативный набор доменных типов, а не только «удобные» страницы.

## I-03 · Разделение проблемы TypeScript и проблемы бандлера

Два класса ошибок визуально идентичны («module not found»), но имеют различную природу и требуют разных решений:

| Уровень | Инструмент | Решение |
|---|---|---|
| TypeScript | `tsc --noEmit` | Типы, `@types`, `// @ts-ignore` |
| Бандлер | Turbopack/webpack | Статический граф зависимостей |

Попытка решить проблему бандлера инструментами TypeScript (и наоборот) приводит к потере итерации. Диагностический вопрос: «эта ошибка возникает на этапе проверки типов или на этапе сборки?» — должен задаваться первым.

## I-04 · `serverExternalPackages` решает не ту проблему

Директива `serverExternalPackages` оптимизирует способ бандлинга существующих модулей (native → require вместо включения в бандл), но не отменяет проверку их существования. Это пример API, чьё название создаёт ложную ментальную модель у разработчика.

Более точное название было бы `serverNativeRequirePackages` или `serverUnbundledPackages`. Осознание семантического разрыва между именем и поведением директивы предотвращает будущие неверные гипотезы.

## I-05 · Extractive summary имеет фундаментальное ограничение по шуму

TF-взвешивание предполагает, что высокочастотные термины документа являются тематически значимыми. Это предположение нарушается, когда документ содержит структурно нетематический текст (библиографии, листинги кода, навигационные элементы), в котором встречаются те же термины, что и в основном тексте.

Фильтрация по regex-паттернам частично решает проблему, но является хрупкой: каждый новый класс шума требует отдельного правила. Архитектурный вывод — extractive NLP без семантического понимания имеет потолок качества, преодолеваемый только переходом к language model.

## I-06 · Пользовательский опыт определяется состоянием загрузки

В инструменте типа «вставь URL — получи результат» воспринимаемое качество продукта в значительной мере определяется поведением интерфейса во время ожидания. Отсутствие индикации состояния делает задержку в 2–3 секунды воспринимаемой как зависание, тогда как спиннер с понятной семантикой снимает это ощущение.

Паттерн проектирования состояний `idle → loading → success/error` должен быть явно реализован до завершения feature, а не добавлен впоследствии как UX-доработка.

## I-07 · localStorage достаточен для MVP-истории без серверной инфраструктуры

Персистентность данных в браузере через localStorage устраняет целый класс инфраструктурных зависимостей: нет необходимости в базе данных, сессионном хранилище или аутентификации. Для одиночного пользователя с единственным устройством это полноценное решение.

Ограничения проявляются только при выходе за границы MVP: синхронизация между устройствами, мультипользовательский доступ, аналитика — всё это требует серверного хранилища. Правильный момент перехода — появление конкретного требования, а не превентивная инфраструктура.

---

## I-08 · Мемоизация актуальна при наличии измеримого cost

`useMemo` имеет смысл применять только тогда, когда вычисление является дорогостоящим относительно частоты ре-рендеров. Генерация clone-промпта — строковые операции над объектом в несколько килобайт — является обоснованным кандидатом: функция нетривиальна, результат не изменяется между рендерами одного результата, зависимость стабильна.

Применение `useMemo` к тривиальным вычислениям вводит когнитивную нагрузку без выгоды и может замедлить рендер из-за overhead мемоизации.
