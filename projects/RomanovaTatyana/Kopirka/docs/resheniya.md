# Решения: Kopirka

Документация технических решений, применённых в процессе разработки. Каждая запись описывает проблему, рассмотренные альтернативы и обоснование выбранного подхода.

## R-01 · Bundler-bypass через `eval('require')`

**Проблема:** Turbopack статически анализирует все `import()` выражения на этапе сборки, включая динамические, обёрнутые в `try/catch`. Playwright отсутствует в `node_modules` production-окружения, что вызывает ошибку сборки вне зависимости от runtime-логики.

**Рассмотренные альтернативы:**
- `await import('playwright')` — не работает: Turbopack разрешает статически
- `// @ts-ignore` + `import()` — устраняет TypeScript-ошибку, но не ошибку бандлера
- `serverExternalPackages: ['playwright']` — не подавляет проверку существования модуля
- Удаление Playwright-функциональности — нарушает требование graceful degradation

**Принятое решение:** Вынести загрузку Playwright в отдельный модуль `lib/playwright-loader.ts`, использующий `eval('require')('playwright')`. Конструкция `eval()` является непрозрачной для статического анализатора бандлера — зависимость не отслеживается на этапе сборки, но корректно разрешается в Node.js-рантайме при наличии модуля.

**Обоснование:** Паттерн `eval('require')` — стандартная техника для optional native dependencies в Node.js-экосистеме. Изоляция в отдельном файле предотвращает случайный импорт из других модулей и централизует единственную точку загрузки.

## R-02 · Экстрактивная суммаризация без LLM

**Проблема:** Требовалось генерировать информативное текстовое резюме без зависимости от внешних API и без необходимости в API-ключах в MVP-версии.

**Рассмотренные альтернативы:**
- OpenAI / Claude API — вводят обязательную зависимость от внешнего сервиса и биллинга
- Локальная LLM (llama.cpp, ollama) — неприемлемые требования к ресурсам для serverless-функции
- Первые N предложений — низкое качество для страниц с вводным boilerplate

**Принятое решение:** Реализация алгоритма TF-взвешивания предложений на основе частотности слов в документе. Компоненты алгоритма:
1. Нормализация лексем через стоп-слова
2. Вычисление term frequency для каждого токена
3. Скоринг предложений как сумма TF-весов токенов
4. Позиционный буст: ×1.8 для первого предложения, ×1.4 для второго
5. Отсев шума: `isCitationNoise()` по regex-паттернам (DOI, ISBN, Retrieved), порог специальных символов для фильтрации кодовых фрагментов

**Обоснование:** Алгоритм детерминирован, работает локально, не требует обучения. Качество достаточно для информационного резюме при тестировании на Wikipedia, GitHub README и эссе. Архитектура допускает замену на LLM-реализацию через единственную точку расширения — функцию `generateSummary` в `lib/summarizer.ts`.

## R-03 · Density-based эвристика извлечения контента

**Проблема:** Страницы имеют разнородную HTML-структуру. Семантические теги (`<article>`, `<main>`) отсутствуют на значительной доле реальных сайтов.

**Рассмотренные альтернативы:**
- Извлечение всего `<body>` — содержит избыточный шум: навигацию, футер, сайдбары
- Readability.js (Mozilla) — дополнительная зависимость, лицензия Apache 2.0
- Только семантические теги — недостаточное покрытие

**Принятое решение:** Каскадная стратегия в `lib/extractor.ts`:
1. Приоритетный поиск по семантическим CSS-селекторам: `article`, `main`, `[role="main"]`, `.post-content`, `.entry-content`, `.article-body`
2. При отсутствии — density heuristic: перебор блочных элементов (`div`, `section`), вычисление метрики `wordCount² / htmlLength` для оценки текстовой плотности
3. Fallback к `<body>` при неудаче эвристики

Предварительно применяется каскадное удаление шумовых элементов через `NOISE_SELECTORS`: навигация, футер, сайдбар, `.references`, `.toc`, рекламные блоки, cookie-баннеры.

**Обоснование:** Двухуровневый подход обеспечивает высокое качество для структурированных страниц и приемлемое качество для нестандартных, не вводя зависимости от обучаемых моделей.

## R-04 · Очистка пустых ссылок в Turndown

**Проблема:** Turndown обрабатывает вложенные элементы рекурсивно изнутри наружу. Кастомное правило `skipImages` возвращает `''` для `<img>`, что приводит к генерации конструкций `[](href)` для родительских `<a>`.

**Принятое решение:** Пост-процессинг сгенерированного markdown через регулярное выражение:

```typescript
.replace(/\[[\s]*\]\([^)]*\)/g, '')
```

Выражение соответствует ссылкам с пустым или пробельным текстом, удаляя их из финального вывода.

**Обоснование:** Модификация правила Turndown для `<a>` потребовала бы учёта контекста дочерних элементов, что нарушает декларативную модель правил. Пост-процессинг является более надёжным и изолированным решением.

## R-05 · Разбиение предложений при отсутствии пробела

**Проблема:** Cheerio `.text()` конкатенирует текстовые узлы без разделителей. HTML-структура вида `</p><h2>` после извлечения даёт строку `...sentence.NextHeading`, которую regex `(?<=[.!?])\s+(?=[A-Z])` не распознаёт как границу предложений.

**Принятое решение:** Изменение квантификатора в разделительном regex с `\s+` (один и более пробелов) на `\s*` (ноль и более пробелов):

```typescript
const re = /(?<=[.!?])\s*(?=[A-Z])/g;
```

**Обоснование:** Условие наличия пробела было артефактом ожиданий, не соответствующих реальной структуре извлечённого текста. Убирая это требование, алгоритм корректно обрабатывает плотные HTML-документы без ущерба для нормальных случаев.

## R-06 · localStorage для персистентной истории

**Проблема:** Требовалось сохранять историю последних запросов между сессиями без серверной инфраструктуры (база данных, сессии, аутентификация).

**Рассмотренные альтернативы:**
- Серверное хранилище (Redis, Postgres) — избыточно для MVP, требует аутентификации
- Cookie — ограничения по объёму данных, сложность управления
- URL-параметры — подходят только для текущей сессии

**Принятое решение:** `localStorage` в Client Component с чтением при монтировании компонента (`useEffect`). История дедуплицируется по URL, ограничена 10 записями, хранится в формате `HistoryEntry[]` (url, title, timestamp).

**Обоснование:** localStorage обеспечивает персистентность без серверной зависимости, соответствует масштабу MVP, не требует пользовательской авторизации.

## R-07 · `useMemo` для генерации Clone-промпта

**Проблема:** Функция `generateClonePrompt()` выполняет нетривиальную строковую обработку (форматирование структуры страницы, обрезка до 6000 символов). Вызов при каждом ре-рендере ResultPanel создавал избыточные вычисления.

**Принятое решение:** Мемоизация результата через `useMemo` с зависимостью от объекта `result`:

```typescript
const clonePrompt = useMemo(() => generateClonePrompt(result), [result]);
```

**Обоснование:** `result` изменяется только при новом запросе пользователя. Мемоизация гарантирует однократный вызов функции на каждый результат скрейпинга вне зависимости от частоты ре-рендеров компонента (переключение вкладок, hover-события).
