# Kopirka — Исследовательский отчёт

---

## 1. Выбранный сервис

**Kopirka** — инструмент для воссоздания веб-сайтов через AI.

🔗 Репозиторий: https://github.com/DanydeDeveloper/kopirka  
🔗 Продакшн: https://kopirka.vercel.app

### Что делает

Пользователь вставляет URL любого сайта — сервис скрейпит страницу, извлекает дизайн-контекст (цвета, шрифты, CSS-переменные, тёмная тема, фреймворк, тип лайаута) и генерирует структурированный промпт для Claude Code. Промпт содержит достаточно контекста, чтобы Claude Code воссоздал проект функционально и визуально — как Next.js приложение с TypeScript и Tailwind CSS.

### Гипотеза

**Воссоздание сайта через AI — это задача с недостаточно хорошим инструментарием.**

Существующие решения (screenshot-to-code, v0.dev, bolt.new) работают либо со скриншотом (теряется семантика), либо требуют ручного описания дизайна. Kopirka берёт живой URL, извлекает машиночитаемый дизайн-контекст прямо из HTML/CSS и передаёт его AI-агенту как структурированное техническое задание. Гипотеза: качество клона растёт, если промпт содержит не только контент, но и извлечённые дизайн-токены.

### Почему интересен с точки зрения moat

Сервис находится на пересечении **веб-парсинга** и **prompt engineering**. Конкуренты либо парсят без AI-интеграции (Firecrawl, Jina Reader), либо генерируют UI без парсинга (v0, Lovable). Копирка закрывает зазор: парсинг → дизайн-контекст → готовый промпт для агента. Подробнее — в разделе 6 (Ров).

| **Tailwind CSS v4** | Построение тёмной UI-схемы, адаптивная вёрстка, кастомные scrollbar-стили |
| **Vercel / CI-CD** | Автодеплой через GitHub push, конфигурация `maxDuration` для serverless функций |
| **Prompt engineering** | Структурирование промпта: дизайн-секция, иерархия заголовков, инструкции для Claude Code как целевой модели |

## 3. Продуктовые модули и покрытие скилами

| Модуль | Что делает | Покрывается скилом |
|---|---|---|
| **URL Validator** (`lib/validate.ts`) | Нормализует и валидирует входной URL, добавляет `https://` если нужно, отклоняет не-HTTP схемы | TypeScript, Next.js Route Handler |
| **Fetcher** (`lib/fetcher.ts`) | HTTP-fetch страницы с fallback на Playwright для JS-рендеренных сайтов; обрабатывает таймаут, 403, редиректы | Node.js fetch API, bundler-bypass через `eval('require')`, Claude Code (отладка Turbopack) |
| **Design Extractor** (`lib/extractor.ts`) | До зачистки HTML извлекает: `theme-color`, CSS-переменные с цветами, Google Fonts, dark mode, Tailwind/Bootstrap, тип лайаута; затем извлекает контент через density heuristic | Cheerio, TypeScript, Claude Code (проектирование `DesignContext`) |
| **Markdown Transformer** (`lib/transformer.ts`) | Конвертирует очищенный HTML в Markdown через Turndown; кастомные правила для img и br; постобработка пустых ссылок | Turndown, regex |
| **Clone Prompt Generator** (`lib/clone-prompt.ts`) | Собирает структурированный промпт: дизайн-бейджи, иерархия заголовков, полный контент, инструкции для Claude Code с целевым стеком (Next.js + TS + Tailwind) | Prompt engineering, TypeScript |
| **API Route** (`app/api/scrape/route.ts`) | POST-эндпоинт, оркестрирует пайплайн, возвращает `ScrapeResult` с дизайн-контекстом | Next.js Route Handler, TypeScript |
| **Clone Panel UI** (`components/ClonePanel.tsx`) | Отображает дизайн-бейджи (dark mode, цвет бренда, фреймворк, шрифты), промпт с preview/expand, кнопки копирования и ссылки на AI-ассистентов | React, Tailwind CSS |
| **History Panel** (`components/HistoryPanel.tsx`) | Сохраняет последние 10 URL в `localStorage`, дедуплицирует, восстанавливает при перезагрузке | React hooks, localStorage API |

## 5. Логика

### Последовательность шагов

1. **Идея и scope** — определили единственный core use case: URL → клон-промпт для Claude Code. Сознательно отказались от crawling нескольких страниц, аутентификации, хранения на сервере.

2. **Пайплайн** — выстроили линейный пайплайн: validate → fetch → extract → transform → generate prompt. Каждый шаг — отдельный модуль в `lib/`, что позволяет заменять компоненты независимо.

3. **Fetcher с fallback** — HTTP-fetch достаточен для большинства сайтов. Playwright подключается как опциональный fallback для JS-рендеренных страниц. Ключевое решение: `eval('require')` для обхода статического анализа Turbopack (подробно — в `docs/resheniya.md`, R-01).

4. **Design context extraction** — главная итерация: перенесли извлечение дизайн-токенов до зачистки style-тегов. Добавили `DesignContext` в тип `ScrapeResult`, пробросили через API.

5. **Промпт для Claude Code** — переписали с "создай HTML-файл" на "создай Next.js проект". Структура: дизайн-контекст → иерархия заголовков → полный контент → требования к стеку и файловой структуре.

6. **UI** — убрали все вкладки (Markdown, JSON, Summary, Links), оставили только Clone Panel. Добавили дизайн-бейджи прямо над промптом.

7. **Деплой** — Vercel подключён к GitHub, автодеплой при пуше в `main`.

### Что было самым сложным

**Статический анализ Turbopack** (см. `docs/kosyaki.md`, K-01–K-03). Playwright — опциональная зависимость, но Turbopack разрешает все `import()` статически на этапе сборки, даже внутри `try/catch`. Три подхода не сработали (`// @ts-ignore`, `serverExternalPackages`, conditional import) — решила конструкция `eval('require')`, непрозрачная для статического анализатора.

**Качество извлечения контента** — нет универсального алгоритма. Wikipedia, GitHub README, новостные статьи, лендинги — каждый класс страниц давал новые артефакты. Итеративно добавляли фильтры шума, density heuristic, regex-постобработку.

**Дизайн-контекст** — сложность в том, что style-теги удаляются в начале пайплайна. Пришлось переставить порядок операций: сначала извлечь дизайн-токены, потом зачищать HTML.

### Открытые вопросы

- Промпт проверялся вручную на нескольких сайтах — нет автоматической метрики качества клона. Как измерить "похожесть" воссозданного проекта на оригинал?
- Playwright не работает в serverless-окружении Vercel без дополнительной конфигурации. Для полной поддержки JS-сайтов нужен внешний браузерный сервис (Browserless, Apify).
- Дизайн-контекст извлекается из inline CSS и атрибутов — внешние CSS-файлы (загруженные через `<link>`) не парсятся. Это ограничивает качество извлечения цветов для сайтов без inline-стилей.
- Для длинных страниц контент обрезается до 8000 символов. Как корректно выбирать "важные" части для обрезки?

## 6. ⭐ Ров (задача со звёздочкой)

### Тип 1 — Данные (Data moat)

**Проблема:** Качество клона зависит от качества извлечённого дизайн-контекста. Сейчас мы берём `theme-color`, CSS-переменные и inline-стили — это поверхностный слой.

**Как строить:** Накапливать базу пар `{URL → структура дизайна}`. С каждым запросом улучшать эвристики: какие CSS-селекторы дают лучший контент, какие паттерны классов маркируют hero-секции, карточки, навигацию. Со временем можно обучить классификатор компонентов (navbar, hero, CTA, pricing block) поверх разметки — без LLM, на чистых CSS-паттернах. Данные накапливаются автоматически с каждым пользовательским запросом; конкурент без этой базы получит худший контекст при тех же алгоритмах.

### Тип 2 — Доверие (Trust moat)

**Проблема:** Пользователь не знает, насколько хорош сгенерированный промпт, пока не отправит его в Claude Code. Неудачный клон = потеря доверия к инструменту.

**Как строить:** Ввести оценку качества промпта до отправки. Метрики, которые можно вычислить локально без AI: полнота дизайн-контекста (сколько токенов извлечено), покрытие контента (% оригинального текста в промпте), обнаруженный фреймворк. Отображать confidence score — пользователь видит "дизайн-контекст: 8/10, контент: полный" и понимает, чего ожидать. Доверие строится через предсказуемость, а не через постфактум.

### Тип 3 — Switching cost (стоимость переключения)

**Проблема:** Текущая версия stateless — нет аккаунта, нет истории на сервере, нет накопленного контекста. Уйти к конкуренту стоит ноль.

**Как строить:** Ввести проектную модель: пользователь создаёт "проект клонирования", куда сохраняются итерации промптов, версии, комментарии ("эта версия лучше передаёт цвета"). История итераций и кастомные настройки (например, "всегда таргетировать Vue вместо Next.js") становятся персональными активами, которые нельзя перенести к конкуренту. Дополнительно: интеграция с Claude Code через MCP — когда инструмент встроен в рабочий процесс разработчика, стоимость отказа от него растёт кратно.
