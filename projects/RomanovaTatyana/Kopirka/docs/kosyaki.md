# Косяки: Kopirka

Документация ошибок, допущенных в процессе разработки. Каждая запись содержит контекст возникновения, симптоматику и последствия.

---

## K-01 · TypeScript не разрешает опциональный модуль

**Контекст:** `lib/fetcher.ts`, функция `fetchWithPlaywright`

**Симптом:**
```
error TS2307: Cannot find module 'playwright' or its corresponding type declarations.
```

**Причина:** TypeScript с настройкой `"moduleResolution": "bundler"` требует наличия модуля в `node_modules` даже для динамического `import()`. Playwright не является обязательной зависимостью и не установлен — компилятор отказывает в сборке вне зависимости от наличия `try/catch`.

**Последствие:** Полная блокировка `tsc --noEmit` и `npm run build`.

## K-02 · Turbopack разрешает `import()` статически

**Контекст:** `lib/fetcher.ts`, попытка использовать `await import('playwright')`

**Симптом:**
```
Module not found: Can't resolve 'playwright'
./lib/fetcher.ts:96:25
```

**Причина:** В отличие от webpack 4/5, Turbopack (бандлер Next.js 16 по умолчанию) выполняет статический анализ всех `import()` выражений ещё на этапе сборки — даже внутри блока `try/catch` и даже при наличии `as any` каста. Попытка пометить импорт через `// @ts-ignore` устраняет ошибку TypeScript, но не ошибку бандлера.

**Последствие:** Production build падал с `Build error occurred` на CI (Vercel).

## K-03 · `serverExternalPackages` не предотвращает статический анализ Turbopack

**Контекст:** `next.config.ts`, попытка добавить `playwright` в `serverExternalPackages`

**Симптом:** Добавление `serverExternalPackages: ['playwright']` не устраняет ошибку сборки.

**Причина:** Директива `serverExternalPackages` инструктирует Next.js использовать нативный `require()` вместо бандлинга, однако не подавляет проверку существования модуля на этапе статического анализа. Модуль всё равно должен физически присутствовать в `node_modules`.

**Последствие:** Неверное предположение о механизме работы `serverExternalPackages` привело к потере итерации.

## K-04 · Пустые ссылки в markdown после удаления изображений

**Контекст:** `lib/transformer.ts`, Turndown-конвертация

**Симптом:** В markdown-выводе появлялись артефакты вида:
```markdown
[](/wiki/File:Question_book-new.svg)
```

**Причина:** Turndown обрабатывает вложенные элементы изнутри наружу. Кастомное правило `skipImages` заменяет `<img>` на пустую строку `''`. Однако родительский элемент `<a>`, содержавший только это изображение, конвертируется по стандартному правилу и генерирует ссылку с пустым текстом: `[](href)`.

**Последствие:** Засорение markdown-вывода бессодержательными конструкциями.

## K-05 · Summary включает библиографические цитаты

**Контекст:** `lib/summarizer.ts`, тестирование на Wikipedia

**Симптом:** Экстрактивное резюме содержало предложения вида:
```
• "Symbiotic Relationships: Pragmatic Acceptance of Data Scraping".
• Retrieved 2015-11-05. ^ "QVC Inc. v.
```

**Причина:** Алгоритм скоринга на основе частотности слов не учитывал семантику предложений. Предложения из раздела References получали высокий скор из-за частого упоминания ключевых терминов в названиях публикаций. Фильтрация по длине (> 30 символов) не отсекала цитаты.

**Последствие:** Резюме было нерелевантным для пользователей, ожидающих тематическое извлечение.

## K-06 · Summary включает фрагменты кода

**Контекст:** `lib/summarizer.ts`, тестирование на GitHub README

**Симптом:**
```
• MCP Connect any MCP-compatible client... { "mcpServers": { "firecrawl-mcp": { "command": "npx"...
```

**Причина:** Текстовый контент GitHub README содержал inline JSON-примеры. После извлечения через Cheerio они оказались в составе `textContent`, который передаётся в суммаризатор. Алгоритм не имел механизма определения кодовых фрагментов.

**Последствие:** Нечитаемые bullet-points в Summary вкладке.

## K-07 · Слияние предложений при отсутствии пробела

**Контекст:** `lib/summarizer.ts`, функция `splitSentences`

**Симптом:** Два смысловых предложения объединялись в одно:
```
...the more wealth you generate.Wealth and PowerMaking wealth is not the only way...
```

**Причина:** Регулярное выражение `(?<=[.!?])\s+(?=[A-Z])` требовало наличия хотя бы одного пробела (`\s+`) между предложениями. В исходном HTML заголовок `<h2>Wealth and Power</h2>` следовал непосредственно за `</p>` без текстового узла-разделителя. После извлечения через `.text()` Cheerio конкатенировал их без пробела.

**Последствие:** Снижение качества разбиения на предложения для страниц с плотной HTML-структурой.

## K-08 · `/dev/stdin` недоступен на Windows

**Контекст:** Тестирование API через `curl | node -e`

**Симптом:**
```
Error: ENOENT: no such file or directory, open 'C:\dev\stdin'
```

**Причина:** В Windows отсутствует псевдофайловая система `/dev/`. Паттерн `require('fs').readFileSync('/dev/stdin', 'utf8')` не работает в среде bash-on-Windows (Git Bash / MSYS2), интерпретирующей `/dev/stdin` как буквальный путь в файловой системе Windows.

**Последствие:** Необходимость переписать тест без piping: использовать `node -e "fetch(...)..."` напрямую.

## K-09 · Конфликт портов при запуске второго dev-сервера

**Контекст:** Попытка запустить `npm run dev --port 3334` при уже работающем сервере

**Симптом:**
```
⨯ Another next dev server is already running.
- Local: http://localhost:3000
- PID: 652
```

**Причина:** Next.js 16 реализует глобальную блокировку dev-сервера через PID-файл. При попытке запустить второй экземпляр процесс завершается с ошибкой, а не переходит на альтернативный порт.

**Последствие:** ECONNREFUSED при обращении к порту 3334 — тест обращался к несуществующему серверу.
