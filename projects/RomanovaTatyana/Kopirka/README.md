# ⚡ Копирка

Инструмент для воссоздания веб-сайтов через ИИ.

🔗 Репозиторий: https://github.com/DanydeDeveloper/kopirka  
🔗 Продакшн: https://kopirka.vercel.app

---

## Быстрый старт

```bash
git clone https://github.com/DanydeDeveloper/kopirka.git
cd kopirka
npm install
npm run dev
```

Открыть: http://localhost:3000

## 1. Описание сервиса

Пользователь вставляет URL любого сайта — Копирка скрейпит страницу, извлекает дизайн-контекст (цвета, шрифты, CSS-переменные, тёмная тема, фреймворк, тип лайаута) и генерирует структурированный промпт для Claude Code. Промпт содержит достаточно контекста, чтобы ИИ воссоздал проект функционально и визуально — как Next.js приложение с TypeScript и Tailwind CSS.

### Гипотеза

**Воссоздание сайта через ИИ — задача с недостаточно хорошим инструментарием.**

Существующие решения (screenshot-to-code, v0.dev, bolt.new) работают либо со скриншотом (теряется семантика), либо требуют ручного описания дизайна. Копирка берёт живой URL, извлекает машиночитаемый дизайн-контекст прямо из HTML/CSS и передаёт его ИИ-агенту как структурированное техническое задание. Гипотеза: качество клона растёт, если промпт содержит не только контент, но и извлечённые дизайн-токены.

### Позиционирование

Сервис находится на пересечении **веб-парсинга** и **составления промптов**. Конкуренты либо парсят без ИИ-интеграции (Firecrawl, Jina Reader), либо генерируют интерфейс без парсинга (v0, Lovable). Копирка закрывает зазор: парсинг → дизайн-контекст → готовый промпт для агента.

## 2. Необходимые навыки для сборки

| Навык | Что делает в контексте проекта |
|---|---|
| **Claude Code** | Основной инструмент разработки: генерация и итерация кода, отладка TypeScript-ошибок бандлера, рефакторинг пайплайна скрейпинга, написание промптов клонирования |
| **Next.js App Router** | Серверные обработчики маршрутов для API (`/api/scrape`), разграничение серверных и клиентских компонентов |
| **TypeScript (строгий режим)** | Проектирование интерфейсов (`ScrapeResult`, `DesignContext`), дискриминированные union-типы для обработки успеха/ошибки |
| **Cheerio (парсинг HTML)** | Обход DOM-дерева, эвристический алгоритм определения основного контента, каскадная система удаления шума |
| **Turndown** | Конвертация HTML → Markdown, кастомные правила, постобработка артефактов |
| **Tailwind CSS v4** | Построение тёмной UI-схемы, адаптивная вёрстка |
| **Vercel / CI-CD** | Автодеплой через GitHub push, конфигурация serverless функций |
| **Составление промптов** | Структурирование промпта: секция дизайна, иерархия заголовков, инструкции для Claude Code как целевой модели |

## 3. Продуктовые модули

| Модуль | Что делает | Покрывается навыком |
|---|---|---|
| **Валидатор URL** (`lib/validate.ts`) | Нормализует и валидирует входной URL, добавляет `https://` если нужно | TypeScript |
| **Загрузчик** (`lib/fetcher.ts`) | HTTP-запрос страницы с запасным вариантом через Playwright для JS-сайтов; обрабатывает таймаут, 403, редиректы | Node.js, обход бандлера через `eval('require')` |
| **Извлечение дизайна** (`lib/extractor.ts`) | До зачистки HTML извлекает: `theme-color`, CSS-переменные с цветами, шрифты, тёмную тему, Tailwind/Bootstrap, тип лайаута | Cheerio, TypeScript |
| **Конвертер в Markdown** (`lib/transformer.ts`) | Конвертирует очищенный HTML в Markdown через Turndown; постобработка пустых ссылок | Turndown |
| **Генератор промпта** (`lib/clone-prompt.ts`) | Собирает структурированный промпт: дизайн-контекст, иерархия заголовков, полный контент, инструкции для Claude Code | Составление промптов, TypeScript |
| **API-маршрут** (`app/api/scrape/route.ts`) | POST-эндпоинт, управляет пайплайном, возвращает результат с дизайн-контекстом | Next.js Route Handler |
| **Панель клонирования** (`components/ClonePanel.tsx`) | Отображает дизайн-бейджи (тёмная тема, цвет бренда, фреймворк, шрифты), промпт с превью и кнопки копирования | React, Tailwind CSS |
| **Панель истории** (`components/HistoryPanel.tsx`) | Сохраняет последние 10 URL в `localStorage`, дедуплицирует, восстанавливает при перезагрузке | React хуки |

## 5. Логика создания

### Последовательность шагов

1. **Идея и рамки** — определили единственный основной сценарий: URL → промпт для клонирования в Claude Code. Сознательно отказались от обхода нескольких страниц, авторизации, хранения на сервере.

2. **Пайплайн** — выстроили линейный пайплайн: валидация → загрузка → извлечение → преобразование → генерация промпта. Каждый шаг — отдельный модуль в `lib/`.

3. **Загрузчик с запасным вариантом** — обычный HTTP-запрос достаточен для большинства сайтов. Playwright подключается как опциональный запасной вариант для JS-сайтов. Ключевое решение: `eval('require')` для обхода статического анализа Turbopack.

4. **Извлечение дизайн-контекста** — главная итерация: перенесли извлечение дизайн-токенов до зачистки тегов стилей. Добавили `DesignContext` в тип результата.

5. **Промпт для Claude Code** — переписали с «создай HTML-файл» на «создай Next.js проект». Структура: дизайн-контекст → иерархия заголовков → полный контент → требования к стеку.

6. **Интерфейс** — убрали все вкладки (Markdown, JSON, Резюме, Ссылки), оставили только панель клонирования. Добавили дизайн-бейджи прямо над промптом.

7. **Деплой** — Vercel подключён к GitHub, автодеплой при пуше в `main`.

### Что было самым сложным

**Статический анализ Turbopack.** Playwright — опциональная зависимость, но Turbopack разрешает все динамические импорты статически на этапе сборки, даже внутри `try/catch`. Три подхода не сработали — решила конструкция `eval('require')`, непрозрачная для статического анализатора.

**Качество извлечения контента** — нет универсального алгоритма. Wikipedia, GitHub README, новостные статьи, лендинги — каждый класс страниц давал новые артефакты. Итеративно добавляли фильтры шума, эвристику плотности текста, постобработку регулярными выражениями.

**Дизайн-контекст** — теги стилей удалялись в начале пайплайна. Пришлось переставить порядок операций: сначала извлечь дизайн-токены, потом зачищать HTML.

### Открытые вопросы

- Промпт проверялся вручную — нет автоматической метрики качества клона. Как измерить «похожесть» воссозданного проекта на оригинал?
- Playwright не работает в serverless-окружении Vercel без дополнительной конфигурации.
- Внешние CSS-файлы не парсятся — только встроенные стили.
- Для длинных страниц контент обрезается до 8000 символов. Как выбирать «важные» части?

## 6. ⭐ Ров (конкурентная защита)

### Данные

Качество клона зависит от качества извлечённого дизайн-контекста. Сейчас берём `theme-color`, CSS-переменные и встроенные стили — это поверхностный слой.

**Как строить:** Накапливать базу пар `{URL → структура дизайна}`. С каждым запросом улучшать эвристики: какие CSS-селекторы дают лучший контент, какие паттерны классов маркируют героя, карточки, навигацию. Со временем — классификатор компонентов поверх разметки без нейросети. Данные накапливаются автоматически; конкурент без этой базы получит худший контекст при тех же алгоритмах.

### Доверие

Пользователь не знает, насколько хорош промпт, пока не отправит его в Claude Code. Неудачный клон = потеря доверия.

**Как строить:** Оценка качества промпта до отправки. Метрики, вычислимые локально: полнота дизайн-контекста, покрытие контента, обнаруженный фреймворк. Отображать показатель уверенности — пользователь видит «дизайн-контекст: 8/10, контент: полный» и понимает, чего ожидать.

### Стоимость переключения

Текущая версия без состояния — нет аккаунта, нет истории на сервере. Уйти к конкуренту стоит ноль.

**Как строить:** Проектная модель: пользователь создаёт «проект клонирования», куда сохраняются итерации промптов, версии, комментарии. История итераций и кастомные настройки (например, «всегда таргетировать Vue вместо Next.js») становятся личными активами, которые нельзя перенести к конкуренту. Дополнительно: интеграция с Claude Code через MCP — когда инструмент встроен в рабочий процесс разработчика, стоимость отказа от него растёт кратно.
