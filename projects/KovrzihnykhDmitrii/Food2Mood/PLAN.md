# Food2Mood — PLAN (рвы на 3–6 месяцев)

Методология: 5 типов рвов (данные / доверие / сеть / switching cost / регуляторка). Ниже детальный план для **трёх** из них — выбраны те, которые реально достижимы за горизонт 3–6 месяцев и которые ядро нашей защиты. По остальным двум (сеть, регуляторка) — короткий комментарий в конце.

---

## Ров 1. ДАННЫЕ (own corpus of tagged dishes + user reactions)

### Гипотеза рва
Чем больше у нас **размеченных блюд** (ингредиенты, методы, таксо-теги) и чем больше **реакций гостей на рекомендации** (view/long_view/purchase/skip), тем точнее наша ранжирующая модель. Этот корпус не воспроизведёт конкурент, который придёт позже, даже с такими же моделями: ему неоткуда взять `(dish → features → user reaction)` в нашей таксономии.

### План на 3–6 месяцев

| Месяц | Что делаем | Метрика |
|---|---|---|
| M1 | Подключить **3 пилотных сети** (сегмент 5–20 точек, с кассой iiko/r_keeper), выгружать меню + чеки еженедельно | ≥3 активных сети; ≥20 000 размеченных блюд в корпусе |
| M2 | Завершить **stage B** оффлайн-LLM разметку мягких тегов (`hearty/light/romantic/comfort_food/snack`). Засесть в `docs/menu_tagger_prompt_v1.txt` и прогнать все 20 000 блюд | ≥ 85% блюд с stage B тегами; coverage каждой оси таксономии ≥70% |
| M3 | Включить **событийный логгер** в пилотных кабинетах (`view / long_view / add_to_cart / purchase_paid`). Идемпотентный ingest уже готов | ≥ 500 000 событий / мес; ≥ 50 000 покупок с известным dish_id |
| M4 | Собрать первый **ranking_dataset_rows.jsonl** (>= 200k строк), тренировать baseline LightGBM поверх детерминированного score | LightGBM обгоняет детерминированный score по NDCG@10 на ≥ 3% в backtest |
| M5 | Дать ресторану **inspector** — инструмент, где он видит распределение своих блюд по таксономии, белые пятна, «блюда без реакций» | ≥ 60% активных клиентов открывают inspector ≥ 1 раз в неделю |
| M6 | Первый **cross-venue transfer**: профиль гостя, собранный на одной сети, работает «из коробки» на второй сети той же категории (с явным согласием гостя) | hit-rate@10 на cold-start в новой сети ≥ hit-rate@10 старого гостя внутри сети минус 20% |

### Что показывает, что ров появляется
- **Прирост качества скоринга на корпусе**, который недоступен конкурентам (NDCG@10 на holdout-события в последние 30 дней).
- **Cost per labeled dish** падает за счёт reuse stage A/B разметки (ниже $0.02 за блюдо на марже).
- **Cross-venue transfer работает** — это то, что может сделать только игрок с ≥ 3 сетями.

### Что мешает
- Кассы отдают данные сырьём и разным форматом (iiko ≠ r_keeper ≠ Poster). Без стабильного Order parser корпус шумит.
- Таксономия может «устареть» (появятся новые форматы блюд, например поке-боулы не описывались). Нужна дисциплина релизов таксо-лока.
- Юридика: явное согласие гостя на cross-venue transfer требует корректного 152-ФЗ + договоров с сетями.

## Ров 2. ДОВЕРИЕ (explainability + endorsement)

### Гипотеза рва
В рекомендациях еды пользователь-Менеджер ресторана **не поверит в чёрный ящик**. Если мы каждое рекомендуемое блюдо можем **разложить на слагаемые** и если у нас есть внешний авторитет (нутрициолог, шеф-повар, диетолог), то ресторан покупает нас, а не конкурента с более «умной» моделью, но без объяснения.

### План на 3–6 месяцев

| Месяц | Что делаем | Метрика |
|---|---|---|
| M1 | Доработать `explain_dish_for_user` до **3 уровней объяснения**: (a) в одну строку для гостя, (b) инженерная раскладка score для аналитика, (c) менеджерский язык «он берёт X, мы предложили Y, потому что Y похож на X по ингредиентам и по времени суток» | у 100% рекомендаций есть все 3 уровня |
| M2 | Привлечь **2 внешних эксперта** — нутрициолог и шеф-консультант. Они проходят выборку из 500 рекомендаций и ставят оценку «безопасно/уместно» | ≥ 90% экспертных «ок» на выборке; договор с обоими на 3 месяца |
| M3 | Ввести **Safety badge** на выдаче: если блюдо помечено экспертом как «безопасно для этого constraints-профиля» — бейдж на UI ресторана | доля выдач с активным бейджем ≥ 40% |
| M4 | Запустить **публичный changelog модели** (раз в месяц пост в TG/медиа): что поменяли в скоринге, как это поменяло NDCG/hit-rate, с примерами | ≥ 1 пост/мес, ≥ 500 уникальных прочтений на пост |
| M5 | Сделать **аудит-лог каждого рекомендационного решения** (`recommendation_debug_sample.csv` в продакшене, а не только в демо) | у ресторана есть ссылка «почему в понедельник рекомендации были такими» — time-to-answer < 5 минут |
| M6 | Получить **endorsement от профильного медиа или отраслевого объединения** (FCSI, ФРиО, «Еда»). Разовая коллаборация / совместный материал | ≥ 1 публикация с нашим логотипом; ≥ 2 упоминания в отраслевых чатах |

### Что показывает, что ров появляется
- Доля сделок, где решающим аргументом становится **explainability** (trackим в CRM после демо-встреч).
- NPS среди менеджеров ресторанов — отдельно по метрике «понимаю, почему выдача такая».
- Не менее 1 отраслевой публикации / упоминания.

### Что мешает
- Нутрициолог и шеф — дорогие часы; нужен разумный scope выборки.
- Риск «объяснение противоречит выдаче» — если мы раскладываем score, а в раскладке `ingredient_match=0.005`, менеджер спросит, почему это блюдо вообще тут. Значит, верхний уровень объяснений нельзя генерировать LLM-ом «красиво», только из честных слагаемых.

## Ров 3. SWITCHING COST (для ресторана)

### Гипотеза рва
Если ресторан один раз подключился и получает от нас **CRM-сегменты, ранжировку для приложения, дашборд меню**, то уйти — значит откатиться в «серую коробку рассылок». Наша задача — чтобы в операционном цикле ресторана наш выход стал частью рутины, а не отдельным инструментом.

### План на 3–6 месяцев

| Месяц | Что делаем | Метрика |
|---|---|---|
| M1 | Готовый **CRM-сегментатор**: экспорт групп гостей в Mindbox / Enkod / GetResponse / Sendsay одной кнопкой. Таксономия → сегменты («любят суп + ходят вечером», «постники», «boost novelty») | ≥ 2 интеграции CRM; ≥ 5 рабочих сегментов у пилотного клиента |
| M2 | **API для ранжирования меню в приложении ресторана** (эндпоинт `/rank?user_id=&menu_id=`) с SLA < 150 ms p95 | 3 приложения сетей дёргают его в проде; ретраи < 0.5% |
| M3 | **Menu planner** — инструмент, который на основе наших тегов подсказывает ресторану: «убрать блюдо X (нет реакций 60 дней)», «ввести блюдо класса `bowl + hearty` (дыра в меню)» | ≥ 1 ресторан принял решение о меню на основе planner-а |
| M4 | **Dashboard в операционке** ресторана: ежедневный дайджест «что произошло у гостей», «какая средняя длина отклика», «каких блюд не хватает» | ≥ 60% активных клиентов открывают дашборд ежедневно |
| M5 | **Интеграция с iiko и r_keeper через нативные коннекторы**, а не csv-экспорт. Чеки прилетают в реальном времени | latency от чека до обновления профиля < 15 минут на p95 |
| M6 | **Retention-loop** «письмо гостю на день рождения с блюдом, которого он ещё не пробовал» — сквозной кейс через нас | конверсия таких писем ≥ 2× базовой рассылки по сети |

### Что показывает, что ров появляется
- **Gross retention** ресторанов ≥ 90% через 6 месяцев после подключения;
- **Expansion revenue**: сети добавляют точки / модули (sales-per-account растёт ежемесячно);
- **Time-to-integrate конкурента > наш**: если нас кто-то обгонит, им надо будет заменить CRM-сегменты + API ранжирования + dashboard — что дольше, чем «поменять провайдера рассылок».

### Что мешает
- Если API упадёт в пике часа пик — ресторан разочаруется и рваная интеграция на iiko ему дороже нашей ценности. SLA критичен.
- Легко скатиться в «сделаем ещё одну свистелку», пока нет устойчивого базового предложения — дисциплина продуктового скоупа.

## Короткие комментарии по остальным рвам

**Сеть (network effect).** В нашем B2B2C кейсе сетевого эффекта между ресторанами-клиентами нет. Есть слабый эффект между гостями («мне понравился этот роллинг, потому что его любят те, с кем у меня похожий профиль») — но это скорее усиление рва **данные**, чем самостоятельный ров. В план не ставим.

**Регуляторка.** Важный fence, но не ров. На горизонте 3–6 месяцев: (а) реестр оператора персональных данных (152-ФЗ); (б) договоры обработки с ресторанами; (в) cookie/consent UI. Это не конкурентное преимущество, а гигиена. В план зашито как compliance-трек, не как ров.

---

## Связка рвов друг с другом

- **Доверие** без объяснений → ресторан не включит cross-venue transfer → нет данных для **данных**-рва.
- **Switching cost** без **доверия** → ресторан будет формально отписан, но психологически «ищет замену» при первой ошибке.
- **Данные** без **switching cost** → конкурент с большей моделью съест кейс, даже если у нас лучше корпус.

Поэтому план синхронный: по всем трём рвам идём параллельно, по месяцам, с явными проверяемыми метриками.
