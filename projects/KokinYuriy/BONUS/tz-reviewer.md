# TZ_Reviewer

Скил для проверки готового ТЗ на полноту и однозначность. Запускается после TZ_Writer — или на чужом ТЗ, которое прилетело от заказчика.

## Когда запускать

- После TZ_Writer перед передачей исполнителю
- Когда получил ТЗ от заказчика и нужно быстро найти дыры
- Когда ТЗ написано давно и нужно убедиться, что оно ещё актуально

## Что делает

Читает ТЗ и проверяет по чеклисту. Возвращает список проблем с указанием где именно и что именно неясно. Не переписывает ТЗ — только указывает на проблемы.

## Чеклист проверки

### Полнота
- [ ] Есть чёткое описание исполнителя и его уровня?
- [ ] Понятна цель задачи (зачем, не только что)?
- [ ] Результат описан конкретно — формат, структура, место сохранения?
- [ ] Все модули/функции перечислены?
- [ ] Out of scope явно указан?

### Однозначность
- [ ] Нет субъективных формулировок («удобно», «быстро», «красиво»)?
- [ ] Каждое требование можно проверить объективно?
- [ ] Edge cases прописаны для нетривиальных сценариев?
- [ ] Нет противоречий между требованиями?

### Критерии приёмки
- [ ] Критерии приёмки есть и они проверяемы?
- [ ] По каждому критерию понятно как именно проверять?
- [ ] Кто принимает работу — указано?

### Исполнимость
- [ ] Исполнитель сможет начать без дополнительных вопросов?
- [ ] Все необходимые доступы и ресурсы указаны?
- [ ] Дедлайн реалистичен (если указан)?

## Выход

```
## Результат проверки ТЗ

### Критические проблемы (блокируют выполнение)
- [Раздел X] — проблема: …

### Замечания (снижают качество)
- [Раздел Y] — проблема: …

### Вопросы к уточнению
- …

### Вывод
Готово к передаче / Требует доработки по пунктам выше
```

Если проблем нет — пишет «ТЗ готово к передаче» и перечисляет что проверено.
