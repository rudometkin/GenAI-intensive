# PRODUCT — О продукте

## О продукте

**КБЖУЙ** — персональный AI-навигатор питания. Принцип: «Приготовь один раз — не думай потом».

Пользователь раз в неделю делает батч-готовку, раскладывает еду по пронумерованным контейнерам (1А, 2Б…). Приложение в каждый момент говорит: какой контейнер взять, где он лежит, как разогреть, что подготовить заранее. AI-агент адаптирует план к отклонениям (пиво в пятницу, ресторан, срыв).

**Суть ценности:** не счётчик калорий постфактум, а система управления питанием от цели до контейнера. Единственное решение, которое управляет физическими контейнерами.

**Стек:** React Native + Expo / FastAPI + PostgreSQL + Redis / Claude API ReAct-агент

## Целевая аудитория

### Основная (Фаза 1–2)

**S1. Молодые профессионалы (20–30 лет)** — хотят результат без усилий. Главная боль: срывы ломают всю систему. Killer feature: адаптация к срывам, план продолжает работать.

**S2. Опытные ЗОЖ-щики (25–40 лет)** — уже считают КБЖУ вручную, тратят 20–40 мин/день. Хотят автоматизировать то, что делают и так. Killer feature: точный autogenerated план с объяснением формул.

### Расширенная (Фаза 3–4)

**S3. Семьи** — главная боль: деньги и отходы. Killer feature: стоимость списка покупок, порции ×N.

**S4. Занятые (предприниматели, врачи)** — нет времени, есть деньги. Killer feature: widget, план перестраивается сам при изменении расписания.

**S5. Новички** — не знают с чего начать. Killer feature: режим «Новичок», геймификация, простые рецепты.

## AI-агрегаторы — какие используем и зачем

### Anthropic Claude API (production)

**Модель:** `claude-sonnet-4-6`

**Зачем:** ReAct-агент — основной AI-движок. Паттерн: system prompt → LLM call → parse tool call → execute → loop.

**Что делает агент:**
- Генерирует персональный план питания (оркестрирует `MealPlannerService` через tool `build_meal_plan`)
- Адаптирует план при отклонениях: «съел пиццу (800 ккал)» → пересчитывает остаток недели
- Отвечает на вопросы пользователя в чат-режиме
- Генерирует задачи подготовки (переложи контейнер 3Б за 12 часов)

**Почему Anthropic, не OpenAI:** лучшее понимание длинного контекста (профиль + план + история), нативный tool-use в SDK, нет ограничений по RPM на релевантных для MVP объёмах.

**Стоимость:** ~$0.003 за генерацию плана (3–5 LLM-вызовов, sonnet pricing). При 1000 DAU → ~$90/мес только на план-генерацию.

---

### Ollama (локально, разработка)

**Модель:** `qwen2.5:14b` (помещается в 12GB VRAM RTX 5070 Ti)

**Зачем:** разработка и тестирование агента без трат на API-ключ. OpenAI-совместимый интерфейс — агент использует тот же код, переключается флагом `USE_LOCAL_LLM=true`.

**Ограничения:** ~1–2 мин на генерацию плана (vs < 30с у Anthropic), качество ответов ниже при сложных сценариях адаптации.

## Парсеры — какие используем и зачем

### Нутрициологическая база знаний (собственная)

**Что:** 20 markdown-файлов в `Кулинария/Принципы заготовок/` — принципы батч-готовки, техники, сценарии заготовок.

**Зачем:** контекст для AI-агента при генерации плана. Агент не изобретает логистику — читает проверенные принципы.

**Формат:** человекочитаемый markdown (не JSON) — чтобы агент мог цитировать и объяснять.

### База рецептов

**Что:** `data/recipes/basic_recipes.json` — структурированные рецепты с КБЖУ, временем готовки, техниками.

**Зачем:** `search_recipes` tool агента — фильтрует рецепты под цель пользователя (дефицит/профицит/поддержание), аллергии, бюджет, время готовки.

**Источник:** вручную кюрированная база. Не парсинг внешних сайтов — контроль качества данных КБЖУ важнее объёма.

### Нутрициологические формулы

**Что:** `data/nutrition/products.json` — КБЖУ на 100г по продуктам. `nutri_service.py` — расчёт BMR (Миффлин–Сан-Жеор) и TDEE.

**Зачем:** `calculate_kbzhu` tool агента. Клинически обоснованные формулы, не «примерные» данные — это ответ на скептицизм S2-сегмента.

## Что не используем и почему

| Инструмент | Почему нет |
|------------|------------|
| Парсинг рецептурных сайтов | Качество КБЖУ непредсказуемо, юридические риски |
| Готовые БД продуктов (Open Food Facts) | Много мусора, сложно нормализовать под нашу модель |
| Fine-tuned модели питания | Overkill для MVP, контекстный промпт решает задачу |
| RAG по рецептурным базам | База рецептов маленькая (<500), прямой поиск по PostgreSQL достаточен |
| Векторные эмбеддинги | Отложено до Фазы 4 — когда база рецептов вырастет до 5k+ |
