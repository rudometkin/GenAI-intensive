# PLAN — Рвы и план на 3–6 месяцев

Ров — это то, что делает переключение на конкурента болезненным или невозможным. Для КБЖУЙ актуальны три: данные, switching cost и доверие.

---

## Ров 1: Данные — персонализация, которая не переносится

### Суть

Чем дольше пользователь, тем точнее план. Агент знает: какие рецепты понравились, когда случаются срывы (пятница, командировки), какой реальный TDEE (не расчётный, а по динамике веса), какие продукты в закупке игнорируются. Эти данные не экспортируются конкурентам — они в нашей модели.

### Почему ров появляется именно здесь

Счётчики калорий хранят историю еды — но не паттерны поведения. Рецептурные приложения — рецепты. Мы единственные, кто знает: контейнер → когда съели → что не доели → что перенесли на следующий день. Это поведенческий граф питания, которого нет ни у кого.

### Что делаем (апрель–июль 2026)

| Месяц | Действие |
|-------|----------|
| Апрель–Май | Запустить MVP (Фаза 1). Начать собирать: `container.eaten_at`, `deviation` (тип, ккал), `shopping_item.checked_at` |
| Май | Добавить `weight_log`: пользователь вводит вес раз в неделю → агент калибрует реальный TDEE |
| Июнь | Хранить `recipe_rating` (неявный: съел/не доел/выбросил). Агент учитывает при следующем плане |
| Июль | Dashboard для пользователя: «Ты ешь по плану 73% дней. В пятницу — 40%. Твой реальный TDEE — 2150 ккал» |

### Метрика появления рва

- **Retention week-4 > 40%** (vs ~12% у счётчиков) — люди не уходят, потому что план уже «знает» их
- **Среднее отклонение КБЖУ от цели на 4-й неделе < среднего на 1-й** — персонализация работает
- **% пользователей, которые обновляют план вручную < 20%** — агент справляется сам

### Что мешает

- Нужны реальные пользователи для накопления данных — закрытый круг (данные → ценность → пользователи)
- Решение: запустить MVP на 20–50 знакомых, собрать данные ещё до широкого запуска

## Ров 2: Switching cost — «Контейнерная система»

### Суть

Пользователь физически маркирует контейнеры (маркер, наклейки). Система хранится не только в приложении — она встроена в его кухню. Перейти на другое приложение = разметить контейнеры заново + объяснить новому приложению расположение холодильника, расписание, способ хранения.

Дополнительный switching cost: **история адаптаций**. Агент знает, что пользователь регулярно отклоняется по пятницам и уже заложил это в план. Конкурент начинает с нуля.

### Что делаем (апрель–июль 2026)

| Месяц | Действие |
|-------|----------|
| Апрель–Май | Запустить управление контейнерами (Фаза 1.7): метки, места хранения, статусы |
| Май–Июнь | **Физическая интеграция:** генерировать PDF-наклейки с QR-кодом контейнера. Пользователь печатает и клеит. Сканирование QR → мгновенное обновление статуса |
| Июнь | Добавить `storage.location` с фото полки — пользователь фотографирует холодильник, агент знает «контейнер 2Б — вторая полка справа» |
| Июль | «Переезд» — пользователь перемещает контейнеры через UI, история остаётся |

### Метрика появления рва

- **% пользователей, использующих физические наклейки > 30%** — это anchor-точка в реальном мире
- **Churn среди пользователей с наклейками vs без** — гипотеза: churn в 2 раза ниже
- **NPS-вопрос на 30-й день: «Насколько сложно было бы перейти на другое приложение?»** — цель: >60% отвечают «сложно» или «очень сложно»

### Что мешает

- Печать наклеек — трение для мобильного пользователя (нет принтера)
- Решение: предложить вариант без принтера — просто маркер + шаблон для написания от руки. PDF-наклейки как опция.

## Ров 3: Доверие — «Прозрачная нутрициология»

### Суть

Основной барьер входа: «Я не верю AI-советам по питанию». Счётчики и рецептурные приложения дают рекомендации без объяснений. Мы показываем: BMR = 1680 ккал (формула Миффлина–Сан-Жеора), TDEE = 2310 ккал (коэффициент 1.375 для умеренной активности), дефицит = -500 ккал для похудения 0.5 кг/нед. Это клинически обоснованные данные, не «алгоритм».

Доверие → сарафанное радио. S2-сегмент (опытные ЗОЖ-щики) — opinion leaders: они расскажут в фитнес-чатах.

### Что делаем (апрель–август 2026)

| Месяц | Действие |
|-------|----------|
| Апрель–Май | Экран профиля: показать BMR, TDEE, формулу, коэффициент активности — всё редактируемо вручную |
| Май | «Почему такой план?» — агент объясняет каждое решение: «Рыба в среду — потому что в профиле указана аллергия на куриный белок» |
| Июнь | Верификация рецептов: нутрициолог-консультант проверяет базу. Бейдж «Проверено нутрициологом» на рецептах |
| Июль–Август | Лонгрид/блог: «Как работает КБЖУЙ изнутри» — публикация формул, методологии, источников. Цель: SEO + доверие tech-аудитории |

### Метрика появления рва

- **% пользователей, которые заходят на экран профиля/формул > 40%** — интерес к прозрачности
- **Упоминания в фитнес-комьюнити (TG-каналы, Reddit-аналоги)** — мониторинг вручную на старте
- **Конверсия из «смотрел» в «зарегистрировался» после страницы о методологии > 15%**

### Что мешает

- Нутрициолог-консультант — нужен бюджет или бартер (консультации в обмен на упоминание)
- SEO работает медленно (3–6 месяцев до трафика)
- Решение на старте: публикации в Telegram, VC.ru, Reddit r/HealthyFood вместо SEO
