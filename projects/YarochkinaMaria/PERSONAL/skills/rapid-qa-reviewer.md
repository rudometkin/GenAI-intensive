# Rapid QA Reviewer

## Статус
Используется сейчас

## Когда триггерится
- MVP готов — нужно проверить перед демо или сдачей
- Подозреваю, что что-то сломано, но не знаю где
- Хочу заранее узнать какие вопросы задаст преподаватель / инвестор / стейкхолдер

**Фразы:** «проверь MVP», «что сломано», «готов ли к демо», «smoke test», «rapid qa», «найди баги»

## Что делает
1. Читает код проекта (Glob + Read + Grep)
2. Проходит по user flow шаг за шагом
3. Проверяет: переходы между страницами, работу компонентов, покрытие edge cases
4. Находит заглушки, которые выглядят как готовый функционал
5. Формирует список: что работает / что заглушено / что сломано
6. Предсказывает неудобные вопросы при демо

## Входные данные
- Путь к директории с проектом (обязательно)
- User flow для проверки (опционально, дефолт: проверяет всё)
- Контекст: кому показываем (преподаватель / инвестор / команда) (опционально)

## Что выдаёт на выходе
Отчёт в трёх колонках: ✅ работает / ⚠️ заглушено / ❌ сломано. Топ-3 приоритетных фикса перед демо. Список вопросов, к которым надо быть готовой.

## Как помогает мне в работе
Снижает тревогу перед любым показом. Особенно ценен для MVP с мокданными — легко пропустить edge case, который сломается прямо во время демо.

## Пример реального использования
Перед демо Settld-клона: проверил все 8 страниц, флажил что `task/[id]` с несуществующим ID показывает корректный fallback, убедился что demo mode загружает данные стабильно.
