# MOAT_SETTLD — Конкурентный ров для клона Settld

> Стратегическая проработка устойчивых конкурентных преимуществ для продукта в категории death admin / bereavement coordination.

---

## 1. Почему для такого продукта moat особенно важен

Технически клон Settld несложен: форма → список организаций → статус-машина → дашборд. Это можно воспроизвести за несколько недель. Конкурент может запустить аналогичный интерфейс, не имея ни одного реального кейса.

Но вот что невозможно скопировать быстро:

- **Доверие в момент горя.** Пользователь приходит в самый уязвимый момент жизни. Он выбирает сервис не по функционалу, а по тому, кому он доверяет. Этот выбор часто идёт через рекомендацию — от больницы, адвоката, похоронного бюро.
- **Операционную экспертизу об организациях.** Знание того, как конкретный банк обрабатывает уведомления, какие документы нужны, за сколько дней отвечают — это опыт тысяч кейсов. Его нельзя скопировать, его нужно накопить.
- **Интеграции с организациями.** Каждая прямая интеграция с банком или госструктурой — это месяцы переговоров, юридических согласований, технической работы. Конкурент начинает с нуля.
- **Распределительные отношения.** Если Settld — рекомендованный партнёр двух крупных сетей похоронных бюро и NHS — конкурент лишён этого канала.

**Вывод:** продукт в категории grief / end-of-life выигрывает не функционалом, а слоем доверия, данных и отношений, который строится годами.

## 2. Типы moat и их релевантность для Settld

### 🟢 Trust moat — **очень высокая релевантность**

Это, вероятно, самый сильный moat в категории. Пользователь выбирает сервис в момент максимального стресса — и делает это один раз в жизни. Рекомендация из уст в уста в этой категории работает исключительно мощно: люди, прошедшие через потерю, советуют сервис следующим.

Бренд «надёжного помощника в самый сложный момент» невозможно построить быстро — он требует настоящих кейсов, партнёрств с уважаемыми организациями (хосписы, благотворительные фонды) и отзывов реальных людей.

### 🟢 Integration moat — **очень высокая релевантность**

Прямые API-интеграции с банками, коммунальными службами и госструктурами — это долгосрочные переговорные отношения. Каждая интеграция ускоряет обработку: вместо PDF по email — автоматическое закрытие счёта. Организации будут встраиваться в процессы с одним партнёром, а не двумя. Это winner-takes-most динамика.

### 🟢 Data moat — **высокая релевантность**

Каждый завершённый кейс — это структурированные данные: какие организации задействованы, сколько времени занимает каждый статус, какие документы нужны, где возникают задержки. Со временем это превращается в «базу поведения организаций» — знание, которое делает каждый следующий кейс проходить быстрее и точнее.

### 🟢 Distribution moat — **высокая релевантность**

Похоронные бюро, солиситоры, хосписы, NHS — это «привратники» рынка. Они общаются с семьёй в первые дни. Если Settld — их рекомендованный инструмент, конкурент лишён главного канала привлечения. Эксклюзивное или предпочтительное партнёрство с несколькими крупными операторами закрывает вход.

### 🟡 Process / Operational moat — **средняя релевантность**

Внутренний «playbook» — знание о каждой из 500+ организаций: какой отдел, какие требования, как эскалировать — накапливается через операционный опыт. Этот актив реален, но труднее монетизировать напрямую. Сила этого moat растёт с масштабом кейсов.

### 🟡 Switching costs (организационная сторона) — **средняя релевантность**

Для конечных пользователей switching costs низкие — каждый кейс уникален. Но для организаций (банков, утилит) — другая история: если они обучили персонал, настроили порталы, выстроили внутренние процессы под Settld — переход к конкуренту болезненен.

### 🟠 Network / Coordination moat — **растущая релевантность**

Двухсторонняя сеть формируется медленно, но устойчиво: больше организаций → ценнее для семей → больше семей → организации обязаны присутствовать. Этот moat пока слабее остальных, но усиливается по мере роста базы.

### 🟠 Regulatory / Expertise moat — **умеренная релевантность**

Знание регуляторики в сфере death admin (probate, tell-us-once, GDPR для чувствительных данных) создаёт барьер для входа. Но это скорее «right-to-operate», а не источник долгосрочного преимущества сам по себе.

## 3. Три самых сильных и реалистичных moat для клона Settld

### Moat #1 — Trust moat: «бренд доверия в момент горя»

**Суть:** Settld становится синонимом надёжной помощи в самый уязвимый момент жизни. Это не просто UX — это позиционирование, тон, партнёры, сертификации.

**Почему важен:** В grief tech пользователь не сравнивает функционал. Он выбирает того, кому доверяет. Один неудачный опыт — не просто отток, это слух, который распространяется в сообществах поддержки.

**Как создаётся:**
- Партнёрства с хосписами, благотворительными фондами (Macmillan, Cruse Bereavement), NHS
- Сертификация от уважаемых организаций (charities, councils)
- Тщательная работа с тоном: UI, уведомления, поддержка — всё должно звучать по-человечески
- Реальные истории и отзывы пользователей
- Нулевой коммерческий шум — интерфейс без рекламы, без upsell в неуместных местах

**Почему сложно скопировать:** Репутацию в grief-сообществах нельзя купить — её нужно заработать через реальные кейсы и рекомендации. Новый игрок приходит без этого капитала.

**Продуктовая механика:** Поле «Как вы узнали о нас?» → трекинг рефералов от партнёров → приоритизация развития партнёрских каналов.

---

### Moat #2 — Integration moat: «прямые каналы с организациями»

**Суть:** Каждая прямая API-интеграция с банком или госструктурой — это ускорение обработки (автоматическое вместо ручного) и барьер для конкурентов, который невозможно обойти за месяц.

**Почему важен:** Организации предпочитают одного партнёра, которого они знают, прошли комплаенс-проверку и встроили в процессы. Вторая интеграция — это бюрократический overhead, который никто не хочет.

**Как создаётся:**
- Начать с top-10 организаций по частоте в кейсах (крупнейшие банки, HMRC, DWP)
- Предложить организациям ценность: аналитика по их SLA, автоматизация вместо ручной обработки, снижение нагрузки на call-центр
- Пройти compliance, security audit, DPA agreement
- Постепенно расширять: 10 → 50 → 200 организаций

**Почему сложно скопировать:** Каждая интеграция — это отдельный контракт, проверки безопасности, технические согласования. При 50 интеграциях конкурент отстаёт минимум на 2–3 года.

**Продуктовая механика:** Org portal с аналитикой → организация видит своё время отклика, backlog, SLA — и не хочет терять эту видимость при уходе к конкуренту.

---

### Moat #3 — Data moat: «интеллект кейсов»

**Суть:** Каждый завершённый кейс — это данные. Какие организации вовлечены, сколько длится каждый статус, где чаще всего застревают, какие документы запрашиваются повторно. Накопленный массив превращается в продуктовую интеллектуальную систему.

**Почему важен:** Через 10 000 кейсов Settld знает: «NatWest в среднем отвечает за 8 дней, а Barclays — за 4, если приложить конкретный документ». Этого не знает ни один конкурент, стартующий с нуля.

**Как создаётся:**
- Фиксировать каждый переход статуса с временными метками (уже заложено в MVP)
- Строить агрегированную аналитику: среднее время обработки по организации, по типу кейса
- Использовать для «умных подсказок» пользователю: «Обычно для Barclays нужно ещё вот это»
- Постепенно переходить к predictive: «Ваш кейс будет закрыт через ~X дней»

**Почему сложно скопировать:** Данные накапливаются только с реальными кейсами. Нет кейсов — нет данных. Нет данных — нет подсказок. Нет подсказок — хуже UX.

**Продуктовая механика:** После каждого завершённого кейса — NPS + структурированный фидбэк → обогащение базы.

## 4. Практический roadmap построения moat

### Месяц 1–2: Закладываем фундамент доверия

| Действие | Продуктовое решение | Метрика успеха | Риски |
|----------|-------------------|----------------|-------|
| Партнёрства с 2–3 funeral homes | Страница «Рекомендован партнёрами» + реферальный tracking | % кейсов с реферальным источником ≥ 15% | Партнёры могут не хотеть исключительности |
| Charity endorsement | Лого на главной, совместные материалы | Рост конверсии с главной на регистрацию | Долгий процесс согласования |
| Tone-of-voice аудит всего UI | Перепись всех текстов с позиции «эмпатии» | Снижение drop-off на шаге 2 wizard | Субъективно — нужны тесты |

### Месяц 2–4: Первые интеграции

| Действие | Продуктовое решение | Метрика успеха | Риски |
|----------|-------------------|----------------|-------|
| API-переговоры с 3 крупнейшими банками | Webhook-прием уведомлений вместо email | Время до ACKNOWLEDGED < 24ч для интегрированных орг | Compliance занимает месяцы |
| Org analytics dashboard v1 | SLA-метрики для org-портала | Ежемесячные активные сессии org > X | Орг может не ценить аналитику |
| Structured data schema для всех кейсов | Стандартизация полей case wizard | % кейсов с полными данными > 80% | Пользователи пропускают поля |

### Месяц 3–6: Data intelligence layer

| Действие | Продуктовое решение | Метрика успеха | Риски |
|----------|-------------------|----------------|-------|
| Аналитика SLA по организациям | Раздел «Типичное время обработки» в карточке орг | Рост satisfaction score пользователей | Организации могут возражать против публикации их SLA |
| Smart prompts в кейсах | «Для [org] обычно нужен ещё [документ]» | Снижение REJECTED по причине «неполные документы» | Требует достаточного объёма кейсов для точности |
| Predictive timeline | «Ожидаемое завершение: ~X дней» | Точность предсказания ± 20% | Ошибки в предсказании подрывают доверие |

## 5. Как moat связан с моим клоном

### Что уже является зачатком moat в MVP

| Элемент MVP | Зачаток какого moat |
|-------------|---------------------|
| **State machine** (6 статусов с временными метками) | **Data moat** — каждый переход фиксирует время. При масштабировании → статистика SLA по организациям |
| **Org portal** с фильтрами и действиями | **Integration moat** — этот портал станет точкой входа для прямых API-интеграций |
| **Organisation registry** (19 организаций с категориями) | **Process moat** — база начинает накапливать знания о каждой организации |
| **Case wizard** со структурированными полями | **Data moat** — стандартизированные данные кейса = основа для аналитики |
| **Demo data** с реальными org-категориями | **Distribution moat** — показывает банкам и утилитам «вот как это выглядит для вас» |

### Что в учебной версии намечено, но стратегически важно

- **История статусов с временными метками** — сейчас просто UX-элемент, но это структура будущей базы данных кейсов
- **Разделение ролей инициатор/организация** — без этой архитектуры нельзя строить B2B-интеграции; в MVP это уже есть
- **Категории организаций** — начало таксономии, которая станет основой knowledge base

### Как клон можно развивать в сторону устойчивого продукта

1. **Добавить аналитику в org portal** → организации начинают зависеть от данных
2. **Добавить document vault** → пользователи хранят документы → sticky engagement
3. **Добавить «typical SLA»** для каждой организации → используем накопленные данные
4. **Партнёрский API** для funeral homes → они встраивают форму в свой процесс → distribution moat
5. **«Verified organization» badge** → организации конкурируют за репутацию на платформе → network effect

## 6. Итог

**Moat-профиль Settld — это триада: Trust + Integration + Data.**

Ни один из этих трёх элементов не работает в одиночку, но вместе они создают почти непробиваемую защиту:

- **Trust** закрывает вход через отношения и репутацию в grief-сообществах — новый игрок приходит «чужим»
- **Integration** закрывает вход через операционные барьеры на стороне организаций — конкурент начинает переговоры с нуля
- **Data** закрывает вход через качество продукта — каждый следующий кейс делает сервис чуть умнее, а конкурент всегда отстаёт

Этот профиль особенно силён потому, что все три источника усиливают друг друга: trust приводит кейсы → кейсы создают данные → данные улучшают продукт → лучший продукт укрепляет trust → trust позволяет вести переговоры с организациями → интеграции ускоряют кейсы → ещё больше кейсов.

**Flywheel запущен — и остановить его очень дорого.**
