# PROJECTCLONE

## Общая логика работы над заданием

Задание потребовало не просто выбрать сервис, а пройти полноценный исследовательский цикл:
осознанно определить критерии → собрать пул кандидатов → оценить системно → сделать
аргументированный выбор. Ниже описан именно этот процесс.

**Шаги:**
1. Определила критерии пригодности сервиса для учебного клонирования.
2. Выбрала рыночные направления, где хочу работать — серебряная экономика и смежные темы.
3. Собрала 20 реальных сервисов-кандидатов, разбила по категориям.
4. Провела сравнительную оценку каждого по 10 критериям.
5. Выделила топ-3 по итоговым баллам.
6. Выбрала финальный сервис и обосновала выбор.
7. Описала продуктовые модули и скиллы, необходимые для реализации клона.

**Критерии пригодности сервиса (использовались при оценке):**

| Критерий | Что проверяла |
|----------|---------------|
| Понятность сценария | Можно ли за 30 сек объяснить, что делает сервис |
| Ясность core value | Насколько очевидна ключевая ценность |
| Декомпозируемость | Выделяются ли 4–7 независимых продуктовых модулей |
| Бизнес-логика | Есть ли роли, статусы, workflow, документы, платежи |
| MVP-реалистичность | Можно ли собрать рабочий клон за ограниченное время |
| Архитектурная описуемость | Можно ли нарисовать ERD, API-схему, компонентную модель |
| Скилл-описуемость | Легко ли перечислить нужные технические и продуктовые навыки |
| Потенциал moat-анализа | Есть ли сетевые эффекты, switching costs, барьеры входа |
| «Умный выбор» для препода | Нестандартность, содержательность, наличие позиции |
| Риск масштаба | Не слишком большой и не слишком примитивный |

### 1.2 Топ-3 финалиста

Оценка по 10 критериям (каждый от 1 до 5, максимум 50 баллов):

| Место | Сервис | Балл | Ключевое преимущество |
|-------|--------|------|----------------------|
| 🥇 1 | **Settld** | 49/50 | Идеальный баланс: чёткий сценарий, богатые модули, реалистичный MVP, сильный moat |
| 🥈 2 | **Everplans** | 47/50 | Богатейшая бизнес-логика: роли, уровни доступа, event-triggered разблокировка |
| 🥉 3 | **Papa** | 44/50 | Классический двусторонний маркетплейс с богатой структурой и marketplace moat |

**Почему Everplans не на первом месте.** Продукт слишком статичен в своём core-сценарии:
пользователь заполняет vault и ждёт события, которое может не наступить годами. Это
усложняет демонстрацию ценности в учебном MVP. Settld активирует продукт немедленно —
в момент реальной острой потребности.

**Почему Papa не на первом месте.** Двусторонний маркетплейс требует двух полноценных
кабинетов, matching-логики и trust/safety механик. За ограниченное время это трудно
сделать содержательно, а не формально.

### 1.3 Финально выбранный сервис: Settld

**Сайт:** [settld.care](https://www.settld.care)  
**Рынок:** Великобритания, B2C (физические лица, занимающиеся делами умершего)  
**Категория:** Death admin / bereavement support / end-of-life administration

#### Что делает сервис

После смерти человека родственники обязаны уведомить десятки организаций — банки,
коммунальные компании, страховщиков, государственные органы, телеком-провайдеров.
Это болезненный, хаотичный и трудоёмкий процесс, происходящий в самый тяжёлый
жизненный момент.

Settld решает это так: пользователь один раз заполняет структурированную форму
с данными умершего → сервис рассылает уведомления в зарегистрированные организации
→ пользователь отслеживает статус каждого запроса в личном кабинете до завершения.

**Ключевые роли:** инициатор (родственник) / организация (bank, utility, etc.)  
**Ключевые объекты:** Death Case, Organization, Notification Request, Status, Document  
**Ключевые события:** создание кейса → отправка запросов → смена статусов → закрытие

#### Почему выбрала именно его

**1. Сценарий кристально чистый.**
Начало (смерть) → процесс (форма + отправка + трекинг) → конец (все организации
уведомлены). У большинства сервисов сценарий размытый. Здесь он завершённый
и однозначный — это редкость.

**2. Продуктовые модули выделяются естественно.**
Без натяжки можно выделить 6 модулей:
- Онбординг и форма смерти (wizard)
- Реестр организаций (справочник с категориями)
- Движок отправки запросов (создание Notification Request per Org)
- Машина состояний (статусы: sent → acknowledged → processing → completed / rejected)
- Дашборд прогресса (сводная картина по всем задачам)
- Уведомления (email/push при изменении статуса)

**3. MVP реализуем за ограниченное время.**
Реальные интеграции с организациями не нужны для учебного клона — достаточно
смоделировать движок запросов и статусную машину. Это убирает самую сложную
техническую часть, сохраняя всю продуктовую глубину.

**4. Moat богатый и неочевидный.**
- *Сетевые эффекты:* чем больше организаций подключено — тем ценнее сервис для
  пользователей; чем больше пользователей — тем выгоднее организациям интегрироваться.
- *Регуляторные партнёрства:* Settld работает с UK Gov и крупными банками — это барьер
  входа, который нельзя скопировать за деньги.
- *Trust как актив:* в момент смерти люди выбирают надёжный сервис, а не дешёвый.
  Переключение на конкурента требует заново выстроить доверие к каждой организации.
- *Данные:* аккумулируется информация о том, как разные организации обрабатывают
  запросы — это операционный актив.

**5. Тема значимая и незаезженная.**
Ни один участник курса не выберет то же самое. Это даёт пространство для
самостоятельного продуктового мышления и честного анализа без шаблонных ответов.

## 2. Какие скиллы нужны для сборки решения

Анализ применительно к Settld. Разбивка по типам скиллов — от исследования до деплоя.

### Исследовательские скиллы

- **Market research** — понять контекст: кто умирает, кто занимается делами,
  какие организации существуют, как сейчас решается проблема без сервиса.
- **Competitive intelligence** — изучить аналоги (Settld, GoodTrust, Everplans,
  UK Government's Tell Us Once service) и понять, где Settld берёт долю.
- **Problem validation** — собрать реальные сценарии: что именно делает родственник
  в первые дни после смерти, какие шаги самые болезненные.
- **Org mapping** — составить реестр типов организаций (banks, utilities, gov, telecom,
  insurance), понять их процессы приёма уведомлений.

### Продуктовые скиллы

- **JTBD-анализ** — основная работа пользователя: «когда мой близкий умирает,
  я хочу закрыть все административные дела, не отслеживая всё вручную».
- **User flows** — проектирование сценариев для двух ролей:
  инициатор (создание кейса → трекинг) и организация (получение запроса → смена статуса).
- **Product decomposition** — декомпозиция на 6 модулей с описанием
  входов, выходов и зависимостей между модулями.
- **Prioritization** — определение MVP-scope: что входит в v1, что откладывается.
  Для Settld: wizard + реестр организаций + статусная машина. Вне MVP: интеграции с API
  организаций, автоматическая рассылка, документы.
- **Statuses & state machine** — проектирование переходов статусов
  (created → sent → acknowledged → processing → completed / rejected / escalated).

### UX-скиллы

- **Reverse engineering интерфейса** — разобрать существующий Settld по скриншотам:
  шаги wizard, структура дашборда, карточка статуса.
- **Role-based UX** — два совершенно разных интерфейса для одного кейса:
  инициатор видит прогресс, организация видит входящий запрос.
- **Wizard design** — проектирование многошаговой формы сбора данных о смерти
  (deceased info, documents, list of organisations to notify).
- **Empty states & edge cases** — что видит пользователь, пока ни одна организация
  ещё не ответила; что происходит при rejected-статусе.

### Технические скиллы

- **Data model** — проектирование ERD:
  `User`, `DeathCase`, `Organization`, `NotificationRequest`,
  `StatusEvent`, `Document`, `Notification`.
- **Backend logic** — реализация создания кейса, привязки организаций,
  генерации запросов, обработки смены статусов.
- **State machine implementation** — статусная машина для `NotificationRequest`:
  валидация переходов, логирование событий.
- **Auth** — разграничение доступа: инициатор видит только свои кейсы;
  организация видит только запросы, адресованные ей.
- **Notifications** — email/push при смене статуса запроса.
- **Frontend** — wizard (multi-step form), dashboard с прогрессом,
  карточки статусов, admin-view для организации.
- **Integrations (mock)** — в учебном клоне интеграции с реальными org-API
  заменяются mock-сервисом, который эмулирует ответы организаций.

### AI skills

- **Prompt engineering** — формулировка задач для кодогенерации:
  описание data model, state machine, UI-компонентов.
- **Multi-step agent workflow** — использование агентов для последовательного
  выполнения задач: исследование → декомпозиция → генерация кода → ревью.
- **Research copilots** — автоматизированный сбор информации
  об организациях, конкурентах, пользовательских сценариях.
- **Code generation** — генерация backend-логики, компонентов, тестов.
- **Code review with AI** — ревью сгенерированного кода на корректность
  бизнес-логики и соответствие data model.

### Delivery-скиллы

- **Testing** — unit-тесты для state machine (ключевая бизнес-логика),
  integration-тесты для wizard flow.
- **Bug fixing** — итерации после первичной сборки.
- **Deployment** — деплой MVP (Vercel / Railway / аналог).
- **Documentation** — описание архитектуры, API-контрактов, data model для README.

### Стратегические скиллы

- **Moat analysis** — структурированный анализ рва:
  сетевые эффекты (org-side), регуляторные партнёрства, trust как барьер,
  операционные данные как актив.
- **Competitive positioning** — почему Settld выигрывает у ручного процесса
  и у Tell Us Once (государственного аналога).
- **Localisation thinking** — как адаптировать модель для других рынков
  (Россия: Госуслуги, МФЦ, банки, ЖКХ).
- **Growth thinking** — как Settld растёт: org-side привлечение (B2B2C),
  партнёрства с похоронными бюро, интеграция в страховые продукты.
