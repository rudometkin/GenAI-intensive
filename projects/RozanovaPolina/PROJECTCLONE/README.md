# PROJECTCLONE — клонирование внешнего сервиса

## 1. Выбранный сервис

Для клонирования я выбрала **Juno Chat** — healthtech-сервис в формате персонального помощника для здоровья.

В рамках задания я сделала проект **ХроноМед**. Это приложение, в котором пользователь:

- регистрируется;
- заполняет медицинский профиль;
- ведет дневник самочувствия;
- видит аналитику по состоянию;
- получает отчет для врача.

Это не копия интерфейса один в один, а клон ключевой продуктовой идеи: помочь человеку структурировать историю здоровья и использовать ее при общении с врачом.

Репозиторий с кодом:

https://github.com/PolyaRoz/chronomed

## 2. Почему я выбрала этот сервис

Я выбрала этот сервис по нескольким причинам.

1. Мне близка сфера здоровья, потому что я биотехнолог. Поэтому мне было интересно делать проект не на случайную тему, а в области, где я понимаю ценность данных, наблюдений и долгосрочного отслеживания состояния.

2. У меня уже есть собственный проект, связанный с питанием. Тема ХроноМеда хорошо с ним пересекается, потому что питание, самочувствие, лекарства, симптомы и образ жизни часто связаны между собой.

3. У меня самой есть генетическое заболевание. Из-за этого при общении с разными врачами приходится каждый раз заново рассказывать историю заболевания, особенности состояния, лекарства и важные детали. Поэтому идея приложения, которое хранит структурированный медицинский профиль и помогает готовить отчет для врача, для меня лично понятна.

4. Healthtech сейчас активно развивается. Люди хотят лучше понимать свое здоровье, а врачи часто ограничены по времени приема. Поэтому сервисы, которые помогают заранее структурировать информацию о пациенте, выглядят актуальными.

## 5. Загруженный код проекта

Код проекта загружен в публичный репозиторий:

Основные файлы проекта:

- `backend/server.py` — сервер, API, авторизация, аналитика и отчеты;
- `backend/database.py` — база данных SQLite;
- `mobile/index.html` — интерфейс приложения;

- `mobile/assets/styles.css` — стили;
- `Dockerfile` — запуск проекта через Docker;
- `render.yaml` — подготовка к публичному деплою;
- `README.md` — инструкция запуска и скриншоты.

## 6. Как запустить проект

Локальный запуск:

```powershell
git clone https://github.com/PolyaRoz/chronomed.git
cd chronomed
python backend\server.py
```

После запуска открыть:

```text
http://127.0.0.1:8000
```

Запуск через Docker:

```powershell
docker build -t chronomed .
docker run -p 8000:8000 chronomed
```

## 7. Логика создания проекта

Сначала была выбрана основная идея: сделать healthtech-приложение, которое помогает человеку вести историю здоровья и готовить понятную информацию для врача.

Потом сервис был разложен на минимальные рабочие модули:

1. регистрация;
2. профиль здоровья;
3. дневник самочувствия;
4. аналитика;
5. отчет для врача.

После этого был сделан backend. Он отвечает за регистрацию, вход, хранение данных, дневниковые записи, аналитику и отчеты.

Затем был собран frontend. В нем есть экран регистрации, экран входа и основное приложение с вкладками: сводка, дневник, отчет и профиль.

В конце проект был подготовлен к проверке: добавлен GitHub-репозиторий, README, скриншоты, Dockerfile и конфигурация для деплоя.

## 8. Что было самым сложным

Самым сложным оказалось сделать так, чтобы приложение можно было проверить не только на моем компьютере.

Сначала приложение работало только локально:

```text
http://127.0.0.1:8000
```

Это удобно для разработки, но не подходит, если проект нужно показать с другого компьютера или из другой сети.

Поэтому пришлось разобраться с вариантами публикации:

- запуск на `0.0.0.0` для локальной сети;
- Dockerfile для упаковки приложения;
- `render.yaml` для деплоя;
- GitHub-репозиторий для проверки кода;
- идея публичного хостинга, чтобы приложение открывалось по ссылке.

Также сложной частью была авторизация. Нужно было сделать не просто форму, а полноценный сценарий: регистрация, пароль, хеширование, сессия и доступ пользователя только к своим данным.

## 10. Задача со звездочкой: как сделать ров вокруг продукта

Ров — это то, что защищает продукт от конкурентов и делает его сложнее повторить.

### 10.1. Ров на данных

Чем дольше пользователь ведет дневник, тем ценнее становится его история:

- симптомы;
- лекарства;
- реакция на лечение;
- триггеры ухудшения;
- динамика перед визитами к врачу.

Конкурент может скопировать интерфейс, но не может быстро скопировать накопленную персональную историю пользователя.

### 10.2. Ров на привычке

Если человек каждый день отмечает состояние, приложение становится частью рутины:

- утром — короткая отметка;
- вечером — дневник;
- перед приемом — отчет врачу;
- после приема — обновление данных.

Когда продукт встроен в привычку, пользователю сложнее перейти на другой сервис.

### 10.3. Ров на медицинской экспертизе

Продукт можно усилить через врачей и медицинских редакторов:

- шаблоны дневников под конкретные заболевания;
- корректные вопросы врачу;
- проверенные материалы;
- осторожные формулировки без постановки диагноза.

Такой продукт будет отличаться от обычного трекера привычек.

### 10.4. Ров на интеграциях

Можно подключить внешние источники данных:

- Apple Health;
- Google Health Connect;
- Garmin;
- лаборатории;
- клиники;
- телемедицина.

Чем больше интеграций, тем выше ценность продукта и тем сложнее его заменить.

### 10.5. Сетевой ров

Если подключить врачей и клиники, появляется ценность для двух сторон:

- пациент ведет данные;
- врач видит динамику;
- клиника наблюдает пациента между приемами.

Так продукт может вырасти из личного дневника в платформу для пациентов и врачей.

### 10.6. Ров на доверии

В медицинских продуктах доверие особенно важно. Его можно строить через:

- безопасность данных;
- прозрачные правила хранения информации;
- медицинскую экспертизу;
- понятное предупреждение, что приложение не заменяет врача;
- стабильную работу.

Если пользователь доверяет приложению свою медицинскую историю, это становится сильным конкурентным преимуществом.
