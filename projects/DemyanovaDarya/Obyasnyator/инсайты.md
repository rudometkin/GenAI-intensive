# Инсайты

Наблюдения, архитектурные решения и выводы, сделанные в ходе разработки.

---

## Почему приняты именно такие архитектурные решения

**Поиск по ключевым словам вместо векторного поиска**

Это был осознанный выбор для первой версии. Векторный поиск требует отдельного хранилища (например, Pinecone или Supabase с расширением pgvector), обученных языковых моделей для преобразования текста в числовые представления и дополнительных затрат на инфраструктуру. Поиск по ключевым словам работает прямо в памяти, ничего дополнительно настраивать не нужно.

На практике такой подход оказался достаточным: когда человек изучает документ, он формулирует вопросы в терминах самого материала. Именно поэтому простое совпадение слов работает хорошо. Семантический поиск понадобится тогда, когда вопрос сформулирован иначе чем текст — это задача для второй версии.

**Хранение данных в браузере вместо базы данных**

Всё состояние приложения — загруженный документ, история диалога, заметки — хранится в браузере пользователя. Это означает полностью stateless-серверную часть: никаких баз данных, никаких пользователей, никакой аутентификации.

Для учебного инструмента это осознанное решение. Конспекты, книги и личные материалы — чувствительные данные. Пользователь не должен задумываться о том, где они хранятся и кто имеет к ним доступ. Очевидное ограничение: нет синхронизации между устройствами.

**Более мощная модель для диалога, упрощённая для генерации**

Чат с тьютором — это живой диалог, где важны точность формулировок, понимание нюансов и способность объяснить сложное просто. Здесь используется более качественная модель.

Квиз, карточки и задачи — это разовая пакетная генерация: запустил, подождал, получил результат. Скорость здесь не критична, но стоимость важна. Упрощённая модель обходится примерно в пять раз дешевле и справляется с задачей полностью.

**Next.js с публикацией на Vercel**

Это идеальная связка для проекта с серверными вызовами к ИИ-модели. Серверные функции Vercel не имеют проблем с холодным стартом для Node.js, статика раздаётся через глобальную сеть, а публикация новых версий происходит автоматически при каждом коммите в основную ветку. Попытка развернуть проект на стандартном хостинге с PHP провалилась сразу — он просто не поддерживает серверный JavaScript.

## Что сработало хорошо

**Трёхколоночная компоновка.** Интуитивна и не перегружена. Документ по центру, навигация слева, тьютор справа. Пользователь не теряется и понимает назначение каждой части.

**Демо-режим.** Критически важен для знакомства с продуктом. Пользователь видит работающий инструмент прямо сейчас, без необходимости что-то загружать. Это снижает барьер входа и помогает понять, нужен ли вообще этот инструмент.

**Кнопки быстрых действий над чатом.** «Квиз», «Карточки», «Практика» всегда на виду. Пользователь не ищет функции в меню и не задаётся вопросом, что умеет приложение.

**Объяснение по выделению.** Самая полезная функция для плотных академических текстов. Выделяешь непонятный абзац — получаешь объяснение именно этого фрагмента в контексте всего документа.

**Синхронизация активного раздела.** Небольшая деталь, которая сильно улучшает ощущение от навигации: боковая панель всегда показывает, где именно в документе ты находишься.

## Подводные камни, о которых стоит знать

**Биллинг Anthropic устроен на уровне аккаунта, не ключа.** Это не очевидно. Можно создать десять новых ключей от одного аккаунта без пополненного баланса — все они вернут одну и ту же ошибку. Проверять нужно баланс аккаунта, а не наличие ключа.

**Аналогичная ситуация с OpenAI.** Новый аккаунт OpenAI тоже имеет ограничения на использование API с нулевым балансом. При переходе на другой провайдер проблема оказалась той же самой.

**Конфликт библиотек с облегчённой серверной средой.** Некоторые Node.js-библиотеки несовместимы с режимом Edge, который Vercel использует по умолчанию. Это не очевидно из документации и выясняется только при первой публикации.

## Уроки для второй версии

**Добавить семантический поиск.** Поиск по ключевым словам хорошо работает, когда пользователь спрашивает в терминах документа. Семантический поиск нужен для вопросов, сформулированных своими словами. Это значительно улучшит качество ответов тьютора.

**Аутентификация и облачное хранение.** Сейчас все данные исчезают при закрытии вкладки. Для серьёзного учебного инструмента нужно сохранять прогресс: пройденные карточки, результаты квизов, заметки. Добавить вход через аккаунт и хранение данных в облаке.

**Отслеживание прогресса.** Показывать пользователю что он уже изучил, какой процент материала охвачен квизом, какие карточки требуют повторения. Интервальное повторение для карточек — классический и доказанно эффективный метод обучения.

**Поддержка нескольких документов.** Сейчас один сеанс работы — один документ. Нужна библиотека загруженных материалов с возможностью переключаться между ними.

**Потоковая передача ответов.** Сейчас ответ тьютора появляется целиком после завершения генерации. Потоковая передача — когда текст появляется постепенно по мере генерации — создаёт ощущение живого диалога и убирает ощущение подвисания при длинных ответах.

## Наблюдения о продукте

**Активное обучение работает лучше пассивного.** Квиз превращает скучное чтение лекции в проверку себя. Карточки дают структуру для повторения. Практические задания создают момент применения знаний. Это не просто «чат с документом» — это цикл: читаешь → проверяешь → повторяешь.

**Язык интерфейса создаёт доверие.** Русскоязычный интерфейс создаёт ощущение что инструмент сделан для конкретной аудитории, а не адаптирован наспех. Тьютор, который отвечает на языке пользователя, снижает когнитивную нагрузку в и без того непростом процессе обучения.

**Ограничение источника — это фича.** Тьютор отвечает только по материалу загруженного документа — не из интернета, не из общих знаний. Это означает, что пользователь доверяет ответам: они основаны именно на том, что он изучает. Для учёбы предсказуемость и точность важнее широты знаний.
