# **Скилл «Тестирование MVP»**

<u>Назначение</u>

Перед тем как показать MVP внешним пользователям, скилл собирает *план тестирования*, который снимает большую часть очевидных багов и заодно формулирует, *что именно мы хотим узнать* по итогам запуска.

<u>Когда триггерится</u>

- команда «составь тест-план», «проверь продукт перед публикацией», «какие edge-кейсы»;
- финальная фаза работы над проектом из блока 3;
- подготовка публичного релиза или презентации.

<u>Что делает</u>

1. **Ручной прогон.** Расписываю ключевые сценарии пошагово, включая *негативные*: невалидный ввод, разрыв сети, таймаут модели.
2. **Edge-кейсы.** Отдельно перечисляю граничные ситуации по каждой подсистеме: пустые поля, эмодзи, экстремальные длины, символы разных алфавитов.
3. **Нагрузка.** Оцениваю, *что произойдёт при 10, 100, 1000 запросах в минуту*, и где лежат узкие места.
4. **UX-наблюдения.** Собираю список вещей, которые нужно наблюдать вживую: где пользователь задерживается, где путается, где отказывается от целевого действия.
5. **Гипотезы.** Формулирую 3–5 гипотез в формате «мы ожидаем X, если увидим Y — делаем Z».

<u>Какой контекст подкладывает</u>

- чек-лист типовых багов веб-приложений (валидация, авторизация, кэш, CORS, rate limits);
- шаблоны формулировок гипотез для пользовательских интервью;
- список инструментов для лёгкой записи наблюдений — Tally, Notion, PostHog.

<u>Формат выхода</u>

- **Чек-лист ручного прогона** с чекбоксами.
- **Таблица edge-кейсов** — «подсистема → кейс → ожидаемое поведение».
- **Гипотезы релиза** — нумерованный список, каждая гипотеза с метрикой проверки.

<u>Ограничения</u>

Скилл заменяет *ручную валидацию*, но не заменяет полноценный QA-процесс с автотестами для зрелого продукта.
