# **Скилл «Написание ТЗ»**

<u>Назначение</u>

Беру идею в формате «хочу сервис, который делает X» и выдаю техническое задание такого качества, которое можно отдать разработчику без повторных созвонов. Главный фокус — *проверяемые критерии приёмки*, чтобы на этапе сдачи не возникало споров «а мы такого не обсуждали».

<u>Когда триггерится</u>

- команда «сделай ТЗ», «напиши спеку», «сформулируй требования»;
- переход от концепции к реализации в любом моём проекте;
- подготовка задачи для внешнего подрядчика или субагента.

<u>Что делает</u>

1. **Цель.** Одно предложение, в котором есть субъект, действие и измеримый результат.
2. **Границы.** Явно перечисляю, *что НЕ входит в задачу*: это сильнее сокращает объём, чем перечисление входящего.
3. **Сценарии.** Для каждого ключевого пользователя описываю путь в формате «условие → действие → результат».
4. **Функциональные требования.** Нумерованный список, каждый пункт — самостоятельное проверяемое утверждение.
5. **Нефункциональные требования.** Производительность, безопасность, локализация, доступность — только то, что релевантно конкретной задаче.
6. **Критерии приёмки.** Формулирую через *наблюдаемое поведение*, а не внутреннюю реализацию: «при нажатии на кнопку X появляется Y».
7. **Зависимости и риски.** Сторонние API, ключи, лимиты, *потенциальные узкие места*.

<u>Какой контекст подкладывает</u>

- шаблон ТЗ с фиксированной структурой разделов;
- набор формулировок для критериев приёмки в стиле Given/When/Then;
- чек-лист нефункциональных требований для веб-сервисов и мобильных приложений;
- список типовых подводных камней: лимиты OpenAI, CORS, хранение персональных данных.

<u>Формат выхода</u>

Markdown-документ с оглавлением, каждая секция — самостоятельный блок, пригодный для копирования в Jira, Linear или Notion.

<u>Ограничения</u>

Скилл не подменяет собой дизайн-макеты и не заменяет архитектурный документ: он описывает *что должно получиться*, а не *как именно это реализовать*.
