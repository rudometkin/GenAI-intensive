# **Скилл «Ров-стратегия»**

<u>Назначение</u>

Беру описание продукта и возвращаю план построения *двух-трёх разнотипных рвов* на горизонт 3–6 месяцев. Каждый ров описывается не лозунгами, а *конкретными действиями, метрикой роста и ограничениями*, которые мешают его выстраивать.

<u>Когда триггерится</u>

- команда «построй ров», «опиши стратегию моата», «что защитит продукт от копирования»;
- задача с пометкой «⭐» из курса — планирование защитного контура вокруг клонированного сервиса;
- подготовка раздела `PLAN.md` для отчётности по проекту.

<u>Что делает</u>

1. **Диагностика.** Определяю, какие типы рвов в принципе применимы к продукту: некоторые ниши исключают, например, регуляторный ров, другие не поддерживают сетевой.
2. **Приоритизация.** Выбираю *два-три типа*, которые достижимы за 3–6 месяцев и дают наибольший эффект.
3. **Конкретизация.** Для каждого выбранного рва описываю:
   - **Что делаем** — перечень конкретных действий и ответственных.
   - **Метрика появления рва** — то, что можно измерить за квартал.
   - **Что мешает** — ресурсы, компетенции, внешние факторы.
   - **Срок** — ближайшая контрольная точка.
4. **Индикаторы разворота.** Описываю, *в каком случае стоит отказаться* от выбранного направления и переключиться на другой тип рва.

<u>Какой контекст подкладывает</u>

- типология рвов: данные, доверие, сеть, switching cost, регуляторика, бренд, эффект масштаба;
- коллекция прикладных кейсов — как *Notion* выстраивал сетевой ров через шаблоны, как *Stripe* — ров доверия через developer experience;
- набор антипаттернов: «ров через секретность», «ров через патенты без продукта», *«ров через ИИ-магию»*.

<u>Формат выхода</u>

Markdown-документ с одним разделом на каждый выбранный ров. Внутри раздела — подсекции «Что делаем», «Метрика», «Риски», «Срок». В конце — сводная таблица.

<u>Ограничения</u>

Скилл формулирует *гипотезы*, а не гарантированные прогнозы. Любой план требует квартального пересмотра по факту собранных данных.
