# REPORT.md — Bloodwise: AI-расшифровка анализов крови

## Что это за продукт

Веб-сервис, который принимает результаты анализов крови (текст или фото бланка) и возвращает подробную расшифровку на человеческом языке: что значит каждый показатель, что выходит за норму, к какому врачу идти.

Целевая аудитория — пациенты, которые получили бланк из лаборатории и не понимают цифры. Не замена врачу, а первый шаг к пониманию своего здоровья.

---

## Что сделано за последние две недели

### Архитектурные решения

**До:** идея без кода — «AI читает анализы».

**Сейчас:** рабочий MVP с тремя слоями:

1. **Input layer** — два способа подачи данных:
   - Текстовое поле (ручной ввод показателей)
   - Загрузка изображения/PDF (передаётся в Claude как base64)

2. **Processing layer** — два уровня обработки:
   - Клиентский парсер (regex по ключевым показателям: Hb, лейкоциты, тромбоциты, СОЭ, глюкоза, ТТГ, холестерин) → генерирует summary-карточки с цветовой индикацией (норма / отклонение / критично)
   - Claude API (claude-sonnet-4) как ядро интерпретации → структурированный промпт с ролью медицинского ассистента

3. **Output layer** — двухуровневый вывод:
   - Summary-карточки с быстрым статусом показателей
   - Полный текст расшифровки от Claude

### Что изменилось в архитектуре

| Компонент | Было | Стало |
|---|---|---|
| Обработка ввода | Только текст | Текст + изображение (base64) |
| Обратная связь пользователю | Нет | Summary-карточки с цветовой индикацией |
| Промпт | Общий | Структурированный: резюме → по показателям → рекомендации |
| UX | Пустая страница | Drag-and-drop, примеры анализов, счётчик символов |

### Что переделали

- Отказались от серверной части на первом этапе — Claude API вызывается напрямую из клиента через fetch. Снижает latency и сложность деплоя для MVP.
- Промпт переписан дважды: первая версия давала слишком медицинский язык (непонятный пациенту), вторая — слишком упрощённый. Текущая версия задаёт явный формат вывода.
- Убрали попытку парсить PDF на клиенте — слишком нестабильно. PDF передаётся как изображение через FileReader.

## Текущая архитектура (схема)

```
Пользователь
    │
    ├── Текст анализа ──────────────────────────────┐
    └── Фото/PDF бланка → base64 → imageData ───────┤
                                                     ▼
                                        Клиентский парсер (regex)
                                                     │
                                            Summary-карточки
                                                     │
                                         Claude API /v1/messages
                                         (system prompt + input)
                                                     │
                                         Текст расшифровки → UI
```

## Что заложено на будущее

### Краткосрочно (1–2 недели)
- Серверный middleware (Node.js/Python) — убрать API-ключ с клиента
- История анализов (localStorage / простая БД)
- PDF-парсинг через pdf.js для извлечения текста перед отправкой в Claude

### Среднесрочно (1–2 месяца)
- Сравнение анализов в динамике (загрузить несколько бланков → тренды)
- Профиль пользователя: пол, возраст → персонализированные нормы
- Поддержка международных форматов (мкмоль, мг/дл и т.д.)
- Экспорт расшифровки в PDF

### Архитектурные долги
- Нет валидации входных данных (пользователь может ввести что угодно)
- Нет rate limiting
- Нет логирования запросов для улучшения промптов

---

## Метрики, которые хотим отслеживать

- Time-to-result (от загрузки до получения расшифровки)
- Процент анализов, где клиентский парсер нашёл хотя бы один показатель
- Retention: возвращаются ли пользователи с новыми анализами
- Доля фото vs текст (понять, какой input удобнее)

---

## Основные технические решения

**Модель:** claude-sonnet-4 — достаточно быстрая и точная для медицинских текстов, понимает русский язык.

**Мультимодальность:** изображение передаётся как base64 напрямую в content array — позволяет обрабатывать фотографии бланков без OCR-препроцессинга.

**Двухуровневый вывод:** regex-парсер даёт мгновенный визуальный сигнал (карточки) ещё до получения ответа от API — снижает воспринимаемую latency.
